# 画像生成プロンプト — 責任境界の例

以下のタイトルを持つ、クリーンなアーキテクチャ例示図を作成してください。

# タイトル

**例: プラットフォームとプロダクトチームの責任境界**

---

# ユースケース

ある企業が、共有の内部プラットフォーム上で複数のプロダクトアプリケーションを運用しています。

- プロダクトチームは顧客向けサービスを所有する
- プラットフォームチームは共有ランタイム機能を所有する
- セキュリティチームはポリシーとアイデンティティ制御を所有する
- データチームは共有イベントと分析契約を所有する

この図では、共有基盤の上で責任がどのように分かれ、どこで受け渡しが発生するかを見せてください。

---

# 構図

次のようなコンポーネントを持つ共有プラットフォームアーキテクチャを描いてください。

- ストアフロントアプリ
- 請求サービス
- サポートサービス
- API ゲートウェイ
- デプロイメント基盤
- ランタイム基盤
- アイデンティティサービス
- ポリシーサービス
- オブザーバビリティ
- イベントストリーム
- 分析モデル

その上に、次の責任領域をオーバーレイしてください。

- プロダクトチーム
- プラットフォームチーム
- セキュリティチーム
- データチーム

---

## 受け渡しとして示すべき点

次のようなハンドオフマーカーを加えてください。

- サービス契約
- ポリシー標準
- スキーマ承認
- インシデントエスカレーション

これにより、単なる所有の色分けではなく、チーム間の接点も見えるようにしてください。

---

# 重要な見せ方

この図では、

- 共有機能にも明確な責任主体があること
- プロダクト領域と共通基盤で責任の性質が異なること
- いくつかのコンポーネントや契約領域では、責任の接点が重要であること

が読み取れるようにしてください。

# テキストのルール

- 短いラベルのみを使うこと
- キャプション、要点、説明注記を入れないこと
- 密度の高いテキストブロックを避けること

---

# 必ず含めること

- 実在しそうなプラットフォームコンポーネントの上に責任境界が示されていること
- 共有機能にも明確なオーナーが見えること
- 実務的でベンダー中立の例であること

---

# 避けること

- 人物、机、画面、室内風景を描くこと
- 図を企業の組織図にしてしまうこと
- すべてのコンポーネントを厳密に 1 つの所有者へ押し込めること
