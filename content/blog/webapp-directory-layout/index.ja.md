Key Components:

- `src/`: Contains all application source code.
  - `app.js`: The entry point of the application, initializes and starts the server.
  - `config/`: Stores configuration files for different environments.
  - `controllers/`: Handles request logic and interacts with services.
  - `middlewares/`: Functions that execute before or after route handlers.
  - `models/`: Defines data structures and interacts with the database.
  - `routes/`: Defines API endpoints and their corresponding handlers.
  - `services/`: Encapsulates business logic and data processing.
  - `utils/`: Provides reusable utility functions.
  - `validations/`: Contains schemas for validating request data.
  - `views/`: Contains template files used to render dynamic HTML content.
  - `public/`: Static assets (e.g., images, CSS, JavaScript).
- `tests/`: Holds test files for different parts of the application.

### Hertz

Hertz is a high-performance, extensible, and minimalist Go web framework.

рЯМР **Official Website**: [cloudwego.io](https://www.cloudwego.io/docs/hertz/)

Hertz supports IDL-based development because it's part of the CloudWeGo ecosystem, which emphasizes high-performance
microservices and service interoperability. The two most commonly used IDLs with Hertz are Thrift and Protocol Buffers.

In Hertz, IDL stands for Interface Definition Language. It's used to define the API's interface -- including service
methods, request and response types, and data structures -- in a way that is language-agnostic and machine-readable.

Why Use IDL with Hertz?

- рЯЪА Auto-generates handler templates, routes, and models
- рЯІ© Ensures consistency across services and clients
- рЯФТ Strong typing and input validation from the start
- рЯФД Makes integration across services and languages easier

[hz layout](https://www.cloudwego.io/docs/hertz/tutorials/toolkit/layout/) shows the structure of the generated code.

рЯУБ **Recommended Structure**:

- `biz/`: Business logic.
  - `handler/`
  - `model/`
  - `router/`
- `idl/`: defined IDL, location can be arbitrary
- `router.go`: User defined routing methods in addition to IDL
- `main.go`: Application entry point.

рЯМР **Official Website**: [djangoproject.com](https://www.djangoproject.com/)

Django encourages a project structure based on the principle of modularity and reusability. It recommends organizing
your code into multiple self-contained apps that each serve a specific purpose. This design pattern, often called "app

When you create a Django project using `django-admin startproject`, you get the basic scaffold:

вФВ   вФФвФАвФА wsgi.py
вФЬвФАвФА apps/                        # Recommended: store all custom apps here

вФВ   вФВ   вФЬвФАвФА admin.py

рЯІ† Design Philosophy

- Each app should be independent and reusable.
- Apps communicate via import or Django signals.
- Business logic is organized into views, models, and optionally services or use-cases.
- Use DjangoвАЩs configuration settings for environment-specific behavior.

вЬЕ Best Practices

- Group reusable apps under apps/
- Use settings modules for environment separation (e.g., settings/dev.py, settings/prod.py)
- Use class-based views for reusability and clarity
- Keep apps small and focused on a single responsibility
- Leverage DjangoвАЩs built-in features: Admin, ORM, middleware

### Flask

Flask is a lightweight WSGI web application framework in Python.

рЯМР **Official Website**: [flask.palletsprojects.com](https://flask.palletsprojects.com/)

Flask is a micro web framework that gives you the flexibility to organize your project however you like. While simple
scripts can live in a single file, for anything beyond a small app, the Flask community recommends using a modular,
scalable layout.

Below is the recommended project structure for a Flask application that follows best practices for maintainability,
scalability, and testability.

рЯУБ Recommended Flask Project Structure

```
your_project/
вФЬвФАвФА app/                         # Application package
вФВ   вФЬвФАвФА __init__.py              # App factory and initialization
вФВ   вФЬвФАвФА routes/                  # Route handlers (views/controllers)
вФВ   вФВ   вФФвФАвФА user_routes.py
вФВ   вФЬвФАвФА models/                  # SQLAlchemy models
вФВ   вФВ   вФФвФАвФА user.py
вФВ   вФЬвФАвФА services/                # Business logic
вФВ   вФВ   вФФвФАвФА user_service.py
вФВ   вФЬвФАвФА templates/               # Jinja2 HTML templates
вФВ   вФЬвФАвФА static/                  # Static assets (CSS, JS, images)
вФВ   вФЬвФАвФА extensions.py            # App extensions (db, login_manager, etc.)
вФВ   вФЬвФАвФА config.py                # App configuration classes
вФВ   вФФвФАвФА utils/                   # Utility functions and helpers
вФВ       вФФвФАвФА helpers.py
вФЬвФАвФА migrations/                  # Database migrations (via Flask-Migrate)
вФЬвФАвФА tests/                       # Unit and integration tests
вФВ   вФФвФАвФА test_user.py
вФЬвФАвФА .env                         # Environment variables
вФЬвФАвФА requirements.txt             # Python dependencies
вФЬвФАвФА run.py                       # Entry point to start the app
вФФвФАвФА README.md
```

рЯІ† Key Concepts

- App Factory Pattern: Define a **create_app()** function in `app/__init__.py` to create and configure the Flask
  application dynamically.
- Blueprints: Organize views into reusable components using Flask blueprints (e.g., one per feature or module).
- Extensions: Initialize Flask extensions (SQLAlchemy, Migrate, LoginManager, etc.) in extensions.py and import them in
  your app.

вЬЕ Best Practices

- Use environment-based configuration (development, testing, production).
- Separate business logic (services) from request-handling (routes).
- Modularize by feature/domain (especially with blueprints).
- Use Flask CLI for development tasks.
- Integrate pytest or unittest for testing.

Spring Boot is a framework that simplifies the development of new Spring applications.

рЯМР **Official Website**: [spring.io](https://spring.io/projects/spring-boot)

Spring Boot follows a convention-over-configuration philosophy and encourages a clean, layered architecture. The
official recommendation is to organize your project by feature or by technical layer using standard Maven/Gradle layout.

Here's a recommended structure that promotes modularity, scalability, and clarity in Spring Boot projects:

рЯУБ Recommended Spring Boot Project Structure

- `src/main/java/{myapplication}/`: Application source code.
- `src/main/resources/`: Configuration files and templates.
  - `application.properties`: App config
  - `static/`: Static assets
  - `templates/`: Thymeleaf templates (if using)
- `src/test/java/`: Test cases.
- `pom.xml` or `build.gradle`: Dependencies and build setup
- `README.md`

Example Java classes.

```
src/main/java/com/example/myapp/
вФЬвФАвФА MyAppApplication.java       # Main app class with @SpringBootApplication
вФЬвФАвФА customer/                   # A domain feature (sub-package)
вФВ   вФЬвФАвФА Customer.java           # Domain model
вФВ   вФЬвФАвФА CustomerRepository.java # Repository
вФВ   вФЬвФАвФА CustomerService.java    # Service
вФВ   вФФвФАвФА CustomerController.java # Web controller
вФФвФАвФА shared/                     # Optional: shared utilities/config
    вФЬвФАвФА config/
    вФВ   вФФвФАвФА AppConfig.java
    вФФвФАвФА exception/
        вФФвФАвФА GlobalExceptionHandler.java
```

рЯУМ Key Points from the Spring Docs:

- Keep the main class in the root package (e.g., `com.example.myapp`).
- Group features by sub-package (e.g., customer, order, billing).
- Spring automatically scans sub-packages, making component discovery seamless.
- This structure avoids having to manually specify `@ComponentScan`.

## Ruby Framework

### Ruby on Rails

Ruby on Rails is a web-application framework that includes everything needed to create database-backed web applications
according to the Model-View-Controller (MVC) pattern.

рЯМР Official Website: [rubyonrails.org](https://rubyonrails.org/)

Ruby on Rails is a full-stack web framework that follows вАЬConvention over ConfigurationвАЭ and вАЬDonвАЩt Repeat YourselfвАЭ
philosophies. Rails projects have a very standardized, opinionated directory layout designed to streamline development
and encourage best practices.

рЯУБ Recommended Ruby on Rails Project Structure

```
your_rails_app/
вФЬвФАвФА app/                       # MVC architecture and core application code
вФВ   вФЬвФАвФА assets/                # CSS, JavaScript, images, fonts
вФВ   вФЬвФАвФА controllers/           # Controllers handle web requests
вФВ   вФЬвФАвФА helpers/               # View helpers
вФВ   вФЬвФАвФА models/                # ActiveRecord models
вФВ   вФЬвФАвФА views/                 # View templates (ERB, Haml, etc.)
вФВ   вФЬвФАвФА channels/              # ActionCable WebSocket channels
вФВ   вФФвФАвФА jobs/                  # Background jobs (ActiveJob)
вФЬвФАвФА bin/                       # Executable scripts (rails, rake, etc.)
вФЬвФАвФА config/                    # Application configuration
вФВ   вФЬвФАвФА environments/          # Environment-specific settings (development, test, production)
вФВ   вФЬвФАвФА initializers/          # Bootstrapping code and third-party gem configs
вФВ   вФЬвФАвФА locales/               # i18n translation files
вФВ   вФЬвФАвФА routes.rb              # Route definitions
вФВ   вФФвФАвФА application.rb         # Main app config
вФЬвФАвФА db/                        # Database schema, migrations, seeds
вФЬвФАвФА lib/                       # Custom libraries and extensions
вФВ   вФЬвФАвФА tasks/                 # Custom rake tasks
вФЬвФАвФА log/                       # Log files
вФЬвФАвФА public/                    # Static files served directly by the web server
вФЬвФАвФА test/ or spec/             # Tests (depending on testing framework, e.g., Minitest or RSpec)
вФЬвФАвФА tmp/                       # Temporary files (cache, pid, sessions)
вФЬвФАвФА vendor/                    # Third-party code and plugins
вФЬвФАвФА Gemfile                    # Project dependencies
вФЬвФАвФА Rakefile                   # Task runner
вФФвФАвФА README.md
```

рЯІ† Design Philosophy

- Follows the MVC pattern: Models, Views, Controllers.
- Convention-driven file and directory layout.
- Emphasizes вАЬfat models, skinny controllersвАЭ вАФ business logic lives mostly in models.
- Built-in generators to scaffold models, controllers, tests, etc.
- Supports RESTful routing by default.

вЬЕ Best Practices

- Organize code in `app/` by function (models, controllers, views).
- Use `lib/` for shared code that doesnвАЩt fit MVC.
- Keep environment-specific configs under `config/environments/`.
- Use migrations to evolve database schema (`db/migrate/`).
- Write tests in `test/` or `spec/` (RSpec).
- Use assets pipeline in `app/assets/` for CSS/JS/images.

## PHP Framework

### Laravel

Laravel is a web application framework with expressive, elegant syntax.

рЯМР Official Website: [laravel.com](https://laravel.com/)

Laravel is a PHP web framework known for its elegant syntax, built-in tools, and emphasis on developer productivity. It
follows the MVC (ModelвАУViewвАУController) pattern and provides a clear, structured project layout out-of-the-box. Its
default directory structure is already well-suited for most applications and considered a best practice by the Laravel
community.

рЯУБ Recommended Laravel Project Structure (Default Best Practice)

```
your-laravel-app/
вФЬвФАвФА app/                      # Core application code (MVC)
вФВ   вФЬвФАвФА Console/              # Artisan commands
вФВ   вФЬвФАвФА Exceptions/           # Custom exception handling
вФВ   вФЬвФАвФА Http/
вФВ   вФВ   вФЬвФАвФА Controllers/      # Controllers for handling routes
вФВ   вФВ   вФЬвФАвФА Middleware/       # HTTP middleware
вФВ   вФВ   вФФвФАвФА Kernel.php        # HTTP kernel
вФВ   вФЬвФАвФА Models/               # Eloquent models (Laravel 8+)
вФВ   вФЬвФАвФА Policies/             # Authorization logic
вФВ   вФЬвФАвФА Providers/            # Service providers (app bootstrapping)
вФВ   вФФвФАвФА Services/             # (Optional) Business logic services
вФЬвФАвФА bootstrap/                # App bootstrapping, including app.php
вФВ   вФФвФАвФА cache/                # Cached files (routes, config, services)
вФЬвФАвФА config/                   # Configuration files (app, db, mail, etc.)
вФЬвФАвФА database/
вФВ   вФЬвФАвФА factories/            # Model factories for testing/seeding
вФВ   вФЬвФАвФА migrations/           # Database migrations
вФВ   вФФвФАвФА seeders/              # Data seeders
вФЬвФАвФА public/                   # Web root, contains index.php
вФЬвФАвФА resources/
вФВ   вФЬвФАвФА js/                   # JavaScript files (Vue, React)
вФВ   вФЬвФАвФА css/                  # Stylesheets
вФВ   вФЬвФАвФА lang/                 # Localization files
вФВ   вФФвФАвФА views/                # Blade view templates
вФЬвФАвФА routes/
вФВ   вФЬвФАвФА web.php               # Web routes
вФВ   вФЬвФАвФА api.php               # API routes
вФВ   вФФвФАвФА console.php           # Console command routes
вФЬвФАвФА storage/                  # Logs, compiled views, file uploads
вФВ   вФЬвФАвФА app/
вФВ   вФЬвФАвФА framework/
вФВ   вФФвФАвФА logs/
вФЬвФАвФА tests/                    # Unit and feature tests (PHPUnit)
вФВ   вФЬвФАвФА Feature/
вФВ   вФФвФАвФА Unit/
вФЬвФАвФА vendor/                   # Composer dependencies
вФЬвФАвФА .env                      # Environment config
вФЬвФАвФА composer.json             # PHP dependencies
вФФвФАвФА artisan                   # Laravel CLI entry point
```

рЯІ† Laravel Philosophy

- вАЬConvention over ConfigurationвАЭ but with flexibility.
- Clean separation of concerns: Controllers, Models, Views, Middleware, Services.
- Encourages RESTful routing and resource controllers.
- Provides first-class testing, migration, validation, and auth scaffolding.

вЬЕ Best Practices

- Use Services or Actions folders (app/Services or app/Actions) for business logic.
- Organize tests under tests/Feature and tests/Unit.
- Use route model binding for cleaner controllers.
- Follow the Single Responsibility Principle (SRP) вАФ thin controllers, fat models/services.
- Use environment configuration for flexibility (.env files).
- Use LaravelвАЩs dependency injection and service container to manage class dependencies.
