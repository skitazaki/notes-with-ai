第5回は、メタデータがクエリ計画の一部になることを説明しました。

エンジンはデータを読む前に、スキーマ、スナップショット、ファイル一覧、統計情報を解決します。この時点でも、メタデータは単なる文書ではないという主張はかなり強いものになっています。しかし AI システムは、その主張をさらに押し広げます。検索パイプライン、特徴量ストア、埋め込みインデックスは、構造だけでは正しく動けません。意味、出所、鮮度、互換性、ポリシーについての機械可読なコンテキストが必要です。

そこで、メタデータは意味インフラになります。

本記事は [メタデータはクエリ計画になる](/ja/blog/metadata/part5-query-planning-metadata/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの実際の問いは、AI アプリケーションがどこかでメタデータを使っているかどうかではありません。それはすでにそうです。問題は、どのメタデータ項目が機械の挙動を直接変え、それらが検索やモデル提供をどう支え、欠けたり古くなったりすると何が壊れるかです。

## 構造メタデータの限界

従来の分析システムは、構造メタデータだけでもかなり遠くまで行けます。

クエリエンジンが列名、データ型、パーティションレイアウト、ファイル統計を知っていれば、スキャン計画、ファイルプルーニング（条件に合わないファイルや範囲を先に除外すること）、コスト推定、関係代数としての正しい結果生成はすでにかなり可能です。より豊かな業務文脈があればさらによくなりますが、コア操作の多くはそれがなくても成立します。

AI ワークロードが違うのは、中心となる操作が違うからです。

ベクトル検索サービスは、「どの行がこの predicate に一致するか」だけを問いません。次のような問いを投げます。

- このベクトルはどの埋め込みモデルが生成したのか
- このチャンクはどの元文書のどの版に由来するのか
- これは日本語なのか英語なのか
- このタスクに使うにはまだ十分に新しいか
- このレコードは、このユーザーやテナントに見せてよいか
- このチャンクは見出しなのか、表の行なのか、本文なのか
- この特徴量はオンライン推論、オフライン学習、その両方のどれに使えるか

これらは表示用の飾りではありません。検索、フィルタ、ランキング、再現性、安全性を左右します。同じ次元数を持つ 2 つのベクトルでも、異なるモデルで作られていれば互換でないかもしれません。似たテキストを含む 2 つのチャンクでも、一方が古い文書版から来ていたり、制限されたソースだったりすれば、利用資格は同じではありません。同名の特徴量でも、鮮度ルールや point-in-time ロジックが違えば、入れ替え可能ではありません。

ここが、従来のメタデータシステムからの主な境界の変化です。

- 構造メタデータは、データがどんな形をしているかを機械に伝える
- 意味メタデータは、そのデータが何を表し、どこから来て、どう使え、何と比較してよいかを機械に伝える

分析の世界では、意味の欠落は誤ったダッシュボードや不自然な join として現れがちです。AI システムでは、同じ欠落が「どのコンテキストを取得するか」「どの特徴量を配信するか」「どのレコードを除外するか」「生成回答を信頼できるソースに基づいて答えられるか」を直接変えます。つまり、メタデータが機械の挙動をそのまま変えます。

## 検索と再利用

検索、ランキング、フィルタ、特徴量の再利用には、スキーマやストレージレイアウトを超えたメタデータが必要です。

典型的な Retrieval-Augmented Generation の流れを考えてみます。システムは製品マニュアル、サポートチケット、ポリシー文書を取り込み、ソースをチャンク化し、埋め込みを計算し、ベクトルを保存し、ユーザー質問に対して候補を取得し、必要に応じて再順位付けし、選ばれた文脈をモデルへ渡します。

この流れをうまく支えるには、生のベクトルだけでは足りません。

コサイン類似度は、表現の近いチャンクを見つける助けになります。しかし、アプリケーションが結果を本当に使えるものにするには、たいてい追加の制御が必要です。

- 現在の製品バージョンに限定する
- 呼び出し元のテナントやリージョン外の文書を除外する
- フォーラム投稿より公式ドキュメントを優先する
- 問いと同じ言語のチャンクに絞る
- 鮮度しきい値より古いチャンクを除外する
- 同じソースページのチャンクが多すぎて、証拠の多様性を潰さないようにする

こうした判断はすべて、検索対象オブジェクトに付いたメタデータに依存します。

特徴量の再利用でも構図は同じです。ストレージ層が違うだけです。モデルチームが必要とするのは、`customer_lifetime_value` という列名だけではありません。その特徴量がどう定義され、誰が所有し、どの上流テーブルに依存し、オンライン配信に使ってよいか、どれだけ新しい必要があり、学習コードと配信コードが同じ変換を適用しているかを知る必要があります。そうした情報は、型シグネチャだけからは信頼して復元できません。

だから AI システムの周囲には、コアストアを取り巻くレジストリや metadata service が増えます。ベクトルインデックスには、フィルタや出所確認のための補助情報が必要です。特徴量ストアには、定義、データ系統、エンティティキー、鮮度期待、offline と online の意味論が必要です。セマンティックレイヤーには、下流システムが「正しい列」ではなく「正しい概念」を要求できるだけの機械可読な業務意味が必要です。

パターンは一貫しています。高価な計算はあとで走りますが、その前にメタデータが候補オブジェクト集合を絞ります。

## 特徴量ストア

特徴量ストアは、意味メタデータが運用そのものになることを示す、もっとも分かりやすい例のひとつです。

データウェアハウスでは、列定義と多少の努力があれば、アナリストがテーブルを理解できることがあります。特徴量プラットフォームでは、それだけではほとんど足りません。プラットフォームは、学習と配信のワークロードをまたいで特徴量を再利用可能にしなければならず、多くの場合、厳しいレイテンシと鮮度制約も伴います。つまり、特徴量を join できるか、配信できるか、信頼できるかを、機械が判断できるだけの具体性を持つメタデータが必要です。

[Feast](https://feast.dev/) や [Hopsworks](https://www.hopsworks.ai/) のようなシステムは、この点をレジストリ、feature view、エンティティ、source definition といった形で明示しています。オブジェクトモデルの細部は違っても、メタデータの負担はほぼ共通です。

実用的な特徴量定義には、少なくとも次の項目が必要です。

- 論理的な特徴量名と所有チーム
- 取得に使うエンティティキー
- 上流のテーブルやストリーム
- 変換定義
- 鮮度または有効期間（time-to-live）の期待
- オフライン学習における point-in-time の意味
- オンライン配信の可否
- データ型と null 許容
- 上流データセットやコードへのデータ系統
- 検索性や統制のためのモデルタグやドメインタグ

簡略化した feature レジストリ風の例を見てみます。

```yaml
feature_view: customer_behavior_v3
owner: growth-ml@example.com
entities:
  - customer_id
source:
  table: analytics.customer_events_hourly
  timestamp_field: event_time
ttl: 48h
online: true
features:
  - name: orders_7d
    type: int64
    description: 直近 7 日間に完了した注文数
  - name: spend_30d_jpy
    type: double
    description: 直近 30 日間に確定した JPY 建て支出合計
  - name: support_ticket_rate_30d
    type: double
    description: 直近 30 日間の完了注文あたりサポート問い合わせ件数
lineage:
  upstream_tables:
    - raw.orders
    - support.tickets
  transform_job: dbt.customer_behavior_features
serving_constraints:
  point_in_time_correct: true
  max_staleness: 2h
```

ここで重要なのは、こうした項目が飾りではないことです。

`ttl`、`max_staleness`、`point_in_time_correct` は、その特徴量が特定のワークロードに使ってよいかを直接左右します。`online: true` は、低レイテンシ配信パスがそれに依存してよいかを決めます。エンティティメタデータは、プラットフォームに「どのキーでリクエストと結び付けるか」を教えます。データ系統情報は、どの上流変更がその特徴量を無効化し、再学習を要求し得るかを運用者に教えます。

意味メタデータが弱いと、特徴量システムは微妙な形で壊れます。

典型例は training-serving skew です。モデルがひとつの解釈で特徴量を学習し、本番配信では別の解釈でその特徴量を受け取ることがあります。オフライン定義では履歴全体への join を使い、オンライン側では現在値スナップショット参照を使っている、といったケースです。構造的にはどちらも `double` かもしれません。しかし意味的には別物です。イベント時刻、materialization のロジック、エンティティマッピング、鮮度について明示的なメタデータがなければ、プラットフォームは一貫性を守れません。

とくに point-in-time の意味は重要です。教師あり学習に使う特徴量は、予測時点で利用可能だった情報だけを反映すべきで、あとから届いた情報を混ぜてはいけません。これは構造の問題ではなく意味の問題です。イベント時刻、公開時刻、ソース遅延、変換ポリシーに依存します。こうした前提をエンコードしていないレジストリは、モデルの正しさを安定して守れません。

だから、特徴量メタデータはノートブックの注釈ではなく、運用インフラとして扱うべきです。その特徴量を再利用できるかどうか自体を決めています。

## 埋め込みとベクトルレコード

埋め込みシステムも、別の形で同じ圧力を生みます。

ベクトルデータベースは、密な数値配列を保持し、近傍検索を効率よく返せます。しかし、そのベクトルが実際の文書、製品、チケット、コードチャンク、画像を表すようになると、アプリケーションは「各ベクトルが何を表し、どう検索に参加してよいか」を説明するメタデータを必要とします。

[Qdrant](https://qdrant.tech/)、[Milvus](https://milvus.io/)、[Weaviate](https://weaviate.io/)、[pgvector](https://github.com/pgvector/pgvector) のようなベクトルストアが、埋め込みと一緒にペイロードや filter field を持つのはそのためです。類似度検索は検索パスの一部にすぎません。本番システムは、ほぼ必ずベクトルスコアの前後でメタデータフィルタを使います。

ひとつのチャンク埋め込みに付くメタデータの具体例を見てみます。

```json
{
  "id": "doc_1842#chunk_07",
  "embedding_model": "text-embedding-3-large",
  "embedding_model_version": "2026-04-15",
  "source_document_id": "docs/product-a/admin-guide",
  "source_document_version": "v2.3",
  "chunk_index": 7,
  "char_start": 12840,
  "char_end": 14410,
  "section_path": ["Administration", "Access Control", "Role Inheritance"],
  "language": "en",
  "tenant_id": "acme-co",
  "visibility": "customer-docs",
  "published_at": "2026-05-10T09:00:00Z",
  "valid_from": "2026-05-10T09:00:00Z",
  "valid_to": null,
  "source_kind": "official-doc",
  "checksum": "sha256:8f3f...",
  "pipeline_run_id": "embed-run-2026-05-11-02",
  "filter_tags": ["rbac", "security", "product-a"]
}
```

これらの項目のいくつかは、検索挙動を直接変えます。

`tenant_id` と `visibility` はアクセス制御フィルタを支えます。`language` は多言語検索の候補集合を絞ります。`source_document_version`、`valid_from`、`valid_to` は、古いコンテキストが現行ソースを上回ってしまうのを防ぎます。`embedding_model` と `embedding_model_version` は、互換性保護に効き、再インデックスや異種エンコーダの混在隔離を可能にします。`section_path` と `source_kind` は、再順位付け処理や answer assembler に、平文テキスト以上の判断材料を与えます。

ここで起きていることは明確です。メタデータは、人間があとで検索結果を見やすくするためだけにあるのではありません。システムが何を取得してよいか、取得結果をどう解釈すべきかを形作っています。

この構造は、クエリエンジンと同じです。

1. まず利用可能なオブジェクト集合を特定する
2. メタデータを使ってその集合をフィルタし、境界付ける
3. そのあとで高コストなスコアリングや配信処理を走らせる
4. 後続システムが何を使ったか分かるよう、出所と系統を残す

こうしたメタデータ項目がなければ、ベクトルインデックスは統制しにくく、誤用しやすいものになります。類似度だけでは、結果が最新か、認可済みか、そもそも問合せベクトルと意味的に比較可能かを判断できません。

## チャンクメタデータ

チャンクメタデータは、とくに独立して扱う価値があります。チャンク化はテキスト分割の細部として軽く見られがちですが、実際には検索モデルの一部だからです。

検索拡張型システムでは、チャンクがインデックス、ランキング、根拠付けの単位になることが多いです。つまり、チャンクに付くメタデータが、再現率、関連性、回答品質に影響します。

最低限、チャンクメタデータには次のような項目が入ることが多いです。

- 元文書の識別子と版
- チャンク位置と境界
- 親セクションや見出しパス
- 文書のタイムスタンプや有効期間
- 言語とロケール
- アクセス方針やテナント範囲
- 本文、API リファレンス、表、FAQ などのコンテンツ種別
- どのパーサや抽出方式で取り出したか

なぜこれらがそんなに重要なのでしょうか。

チャンク境界が意味を変えるからです。`Deprecated behavior` という見出しの下にある段落は、語彙が重なっていても、`Current default configuration` の下の段落と同じように扱うべきではありません。表セルは、ヘッダー文脈がなければ理解不能かもしれません。古いリリースノートから取ったチャンクは、話題としては近くても、答えとしては不適切かもしれません。

そのため、チャンクメタデータは3段階で効きます。

第一に、検索フィルタを改善します。システムはスコアリング前に、現行製品版、適切な言語、公式ソースを優先できます。

第二に、ランキングと再順位付けを改善します。モデルやヒューリスティックは、すべてのチャンクをフラットなテキストとして扱うのではなく、セクションタイトル、ソース種別、公開時刻を追加シグナルとして使えます。

第三に、根拠付けを改善します。回答があるチャンクを引用するとき、アプリケーションは曖昧なテキスト塊ではなく、どの文書のどのセクションのどの版かまでリンクできます。

だから、チャンクメタデータは装飾ではなく、検索の実行コンテキストと考える方が実態に合います。retriever が何を候補にしてよく、generator が何を証拠として引用してよいかを定義しています。

この原理はマルチモーダルでも同じです。画像埋め込みには領域座標、撮影時刻、元アセット識別子、モデレーション分類が必要かもしれません。音声チャンクには話者識別、時刻範囲、書き起こしの信頼度が必要かもしれません。モダリティは変わっても、意味メタデータの必要性は変わりません。

## データの出所と系統管理

検索対象オブジェクトがモデル挙動の入力になると、出所情報は任意ではなくなります。

ここでは、`provenance` と `lineage` を分けて考えると整理しやすくなります。出所情報（provenance）は、その検索対象や特徴量が「どの元データ・元文書から来たのか」を示します。系統情報あるいは系統管理（lineage）は、そこから「どの変換、依存関係、公開処理を経て現在の成果物に至ったのか」を示します。前者は源泉の確認、後者は経路と依存関係の確認です。

分析システムでは、系統情報はダッシュボードの項目がどこから来たか、どのジョブがどのソースに依存するかを説明するために使われます。AI システムでは、それに加えて、ある特徴量、埋め込み、検索チャンクがどう作られ、まだ信頼してよいかを説明しなければなりません。

通常、少なくとも次の関係を追跡する必要があります。

- どの元オブジェクトがこの検索レコードを生んだか
- どのパースやチャンク化の工程がそれを変換したか
- どのモデルが埋め込みを生成したか
- どのインデックス構築やマテリアライズ処理がそれを公開したか
- どの下流回答、特徴量ベクトル、推論がそれを消費したか

[OpenLineage](https://openlineage.io/) や [Marquez](https://marquezproject.ai/) は、データシステムにおける一般的な系統情報パターンを示しています。AI の検索パイプラインが必要とするのは、それを置き換えることではなく、そこへ拡張することです。

具体的な失敗例を考えてみます。あるドキュメントページが更新されて、セキュリティ手順が修正されたとします。しかし、ベクトルインデックスには古い版の埋め込みが残っています。ユーザーが正しい設定手順を尋ねると、検索コンポーネントは意味空間的に近い、古い版由来のチャンクを返してしまうかもしれません。もしシステムに文書バージョンの出所情報や、どのパイプライン実行がどのインデックスを生成したかという系統情報がなければ、運用者は「どの古いソースが使われたか」「どの再インデックス実行が失敗したのか」を証明しにくくなります。

適切なメタデータがあれば、プラットフォームは次のような問いに答えられます。

- 文書バージョン `v2.2` から導かれたチャンクはどれか
- エンコーダバージョン `2026-04-15` で生成された埋め込みはどれか
- 現在のインデックス shard をマテリアライズしたパイプライン実行はどれか
- 後から無効化されたチャンクを引用していた回答はどれか

これらは事後分析のぜいたくではありません。ロールバック、選択的な再インデックス、監査性、信頼性を支えます。

系統情報は、検索品質とプラットフォーム運用をつなぐ橋でもあります。埋め込みがパーサ出力、ソースコンテンツ、モデルバージョンの下流成果物であるなら、そのいずれかが変わったとき再インデックスが必要になり得ます。依存グラフを自動化できる程度に明示するのが、メタデータの役割です。

## セマンティックレイヤー

レジストリやベクトルペイロードの先にある次の段階は、機械可読なセマンティックレイヤーです。

分析では、セマンティックレイヤーは指標、エンティティ、業務定義の観点から語られることが多いです。同じ考え方は AI ワークフローでも重要です。モデルや検索システムも、生のフィールドだけでなく、安定した概念を必要とするからです。

たとえば、あるチームが `arr` と呼び、別のチームが `annual_recurring_revenue` と呼び、さらに別のチームが除外条件付きの契約売上を検索対象文書として持っているとします。人間のアナリストなら、文書を読んで最終的に整合を取れるかもしれません。しかし機械パイプラインは、そのような人手解釈を前提にできません。エンティティ、メジャー、スコープ、除外条件の定義が明示されている必要があります。

だから、機械可読な業務意味が価値を持ちます。

セマンティックレイヤーは、たとえば次を定義できます。

- どのエンティティがシステム横断で標準的か
- どの指標が互換で、どれが意図的に違うか
- フィルタやグルーピングに使うべきディメンションはどれか
- どの同義語が同じ業務概念に対応するか
- 各概念について、どのソースが権威あるものか

[dbt Semantic Layer](https://docs.getdbt.com/docs/use-dbt-semantic-layer/dbt-sl) のようなシステムは、分析におけるこのパターンをよく示しています。AI ワークフローでは、同じ種類のメタデータが、検索拡張、概念の整合、特徴量発見、安全なツール選択を支えます。

たとえば、モデル向けシステムが「日本の enterprise 顧客の churn risk を見せてほしい」というリクエストを受けたとき、`enterprise` がどの segment code か、`Japan` が請求国なのか営業地域なのか、`churn risk` が分類スコアなのか手作業定義のヒューリスティックなのかを、毎回推測させるべきではありません。セマンティックレイヤーがあれば、機械が正しい特徴量を取得し、正しいフィルタを組み立て、結果の出所と系統を説明できる程度に、意味を明示できます。

重要なのは、ここでの目的が「カタログを人間が探しやすくすること」だけではない点です。目的は、自動化された検索や判断パスにおける曖昧さを減らす、機械解釈可能な意味です。

## 失敗モード

意味メタデータが能動的に作用するインフラになると、その失敗モードも運用リスクになります。

もっとも分かりやすいのは古い埋め込みです。ソース文書が変わったのに、システムが影響チャンクを再埋め込みしなければ、アプリケーションコードが変わらなくても検索品質は落ちます。ベクトル値は数値としては妥当でも、現在のソース真実を表さなくなります。

壊れた出所情報と系統情報もよくある問題です。チャンクはベクトルインデックス内に存在していても、どのソース版、どのパーサ実行、どの埋め込みジョブに由来するかを追えないことがあります。そうなると、選択的な修復は難しくなり、監査性が必要な回答では信頼も損なわれます。

特徴量定義の不整合は別種の欠陥を生みます。あるチームが同じ特徴量名を再利用しつつ、時間窓、分母、join ロジックを気付きにくい形で変えるかもしれません。モデルは依然として数値入力を受け取りますが、その数値の意味は変わっています。レジストリメタデータと系統情報がなければ、その欠陥は「説明のつかないモデルドリフト」に見え、契約破りだと気付きにくくなります。

ポリシーメタデータの欠落も同じくらい重大です。テナント範囲、法域、可視性ラベルを持たない検索対象は、技術的には取得可能で、意味的にも関連があっても、運用上は不適格かもしれません。類似度検索は、コンプライアンス境界を知りません。メタデータがそれをエンコードしている場合だけ知れます。

意味メタデータでしか防げない互換性失敗もあります。

- 互換でないエンコーダバージョンの埋め込みを混在させること
- イベント時刻ロジックが揃っていないオンライン特徴量とオフラインラベルを結合すること
- すでに新しい版に置き換わったり撤回されたりした文書からチャンクを検索すること
- 別のエンティティ粒度向けに作られた特徴量を誤って別モデルに適用すること

共通するのは、バイトは読めて、型も合って見えるのに、意味、出所、利用制約が十分にエンコードされていないためにシステムが壊れることです。

だから、AI 向けメタデータは、スキーマ進化やテーブルバージョンメタデータと同じ真剣さで保守すべきです。単に探しやすさを改善するのではなく、正しさの境界を制御しています。

## アクティブな制御レイヤー

このシリーズのここまでの流れを振り返ると、パターンは一貫しています。

メタデータは列名と型から始まり、互換性契約になり、物理最適化データになり、レイクハウスの協調状態になり、クエリ計画入力になりました。AI インフラは、その延長線上にあります。システムは依然としてスキーマやレイアウトメタデータを必要としますが、それに加えて、機械が直接行動に使える意味コンテキストも必要になります。

特徴量ストアは、ある特徴量が再利用可能か、新鮮か、point-in-time 的に正しいかをメタデータで判断します。ベクトルデータベースは、どの埋め込みが検索対象として適格かをメタデータで判断します。RAG パイプラインは、セクション文脈、有効期間、ソース識別をメタデータで保ちます。系統管理システムは、ソース変更を再インデックスや監査パスへ結び付けます。セマンティックレイヤーは、業務概念を機械向けに明示します。

この段階になると、メタデータはシステムの横に置かれた受動的な説明ではありません。システムの内側の制御ロジックの一部です。

実務上の帰結は単純です。AI プラットフォームを作るチームは、意味メタデータを第一級のインフラとして扱うべきです。バージョン管理し、検索可能にし、検証し、再インデックス、アクセス制御、特徴量配信、監査といった運用フローへつなぐ必要があります。雑に扱うと、悪くなるのは見つけやすさだけではありません。検索の誤り、特徴量の不整合、本番挙動に対する弱い出所説明が生まれます。

## コントロールプレーンへ

AI ワークロードがより豊かなメタデータを必要とするのは、その中心的な操作が型付き構造だけでなく、意味、データの出所、データ系統、検索コンテキストに依存するからです。

特徴量ストアは、特徴量がどう定義され、どのキーで引かれ、どの頻度で更新され、学習と配信のあいだでどう整合しているかを知る必要があります。ベクトルインデックスは、ベクトルがどの元オブジェクトから来て、どのモデルが生成し、どんなフィルタとポリシーが適用され、いまも有効かを知る必要があります。チャンク化された検索システムは、ランキングと根拠付けを信頼できるものにするため、十分な文脈を保持しなければなりません。セマンティックレイヤーは、名前から推測するのではなく、機械が正しい概念を要求できるようにする必要があります。

だから、意味メタデータは discoverability を改善するだけではありません。AI システムに「何が比較できるか」「何が検索できるか」「何が再利用できるか」「何が信頼できるか」を教えます。

第7回では、このコントロールプレーンの含意を明示します。互換性、性能、計画、意味、ポリシーをメタデータが横断的に調停するようになると、それは実質的にコントロールプレーンです。
