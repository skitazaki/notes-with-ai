第1回は、重要な境界で終わりました。フィールドの意味が明示されると曖昧さは減りますが、協調の仕事がなくなるわけではありません。

チームは、`customer_id` が文字列であり、`created_at` が UTC であり、`plan` が小さな enum に属すると文書化できます。それだけでも検証と解釈は改善します。より難しい問題は、複数の提供者と利用者が同時に同じ形に依存し始めたときに始まります。フィールド定義は、もはや説明のためだけのものではありません。変更を制約し始めます。

その瞬間に、データ型は契約になります。

本記事は [列名とスキーマ](/ja/blog/metadata/part1-field-schema/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの焦点は、一般的なガバナンスの文脈が示しがちな範囲よりもずっと狭いものです。実際の問いは、契約が抽象的によい考えかどうかではありません。ストリーム、サービス、データウェアハウス、ジョブが同じスキーマに依存し始めたとき、スキーマ変更がどう振る舞うのかという点です。

## 契約化の境界

ひとつのスクリプトの中では、型注釈はおおむねローカルなものです。フィールド名を変えたら、同じコードベースを直して終わりです。

システムをまたぐと、同じ変更でも意味が変わります。ある提供側が Kafka にイベントを発行し、ストリーム処理がそれを変換し、ストレージ用ジョブがオブジェクトストレージに書き出し、さらに複数の利用側が同じトピックからビューを作ったりワークフローを起動したりするかもしれません。そうしたシステムは、足並みをそろえて同時にデプロイされるわけではありません。

たとえば、注文サービスが次のイベントを発行するとします。

```json
{
  "order_id": "o-1001",
  "customer_id": "c-42",
  "status": "paid",
  "amount_jpy": 12000,
  "created_at": "2026-05-20T08:15:00Z"
}
```

このペイロードは、データ値以上のものを運んでいます。

- 利用側は `order_id` が安定した識別子だと仮定します。
- 課金システムは、`amount_jpy` が主要通貨単位の小数ではなく、円単位の整数だと仮定します。
- ステータスを扱うコードは、`paid` や `cancelled` のような値に特定の業務上の意味があると仮定します。
- 監視やアラートのロジックは、`created_at` が処理時刻ではなくイベント時刻だと仮定します。

こうした前提が複数の場所に書き込まれた瞬間、そのスキーマは独立して変化するシステム間のインターフェースになります。どんな変更でも、ひとつの具体的な運用上の問いに答えなければなりません。まだ変更していない利用側に何が起きるのか、という問いです。

契約が守るのはそこです。契約は、業務上の意味が正しいことまでは保証しません。提供者と利用者が同時デプロイに頼らずに変更を調停できることを保証します。

## 互換性

ほとんどのスキーマの議論は、最終的に互換性ルールへ収束します。変更された提供側が未変更の利用側と安全にやり取りできるか、あるいはその逆が可能かを決めるのがそのルールだからです。

よく使われる用語は、後方互換性、前方互換性、完全互換性の3つです。言葉だけ見ると単純ですが、実際の提供者と利用者の関係に落として初めて意味を持ちます。

### 後方互換性

後方互換性とは、新しい利用側が古いスキーマで生成されたデータを読めることです。

まず、単純な Avro 風のレコードを見てみます。

```avdl
record OrderCreated {
  string order_id;
  string customer_id;
  int amount_jpy;
}
```

ここに、デフォルト付きの任意フィールドを追加するとします。

```avdl
record OrderCreated {
  string order_id;
  string customer_id;
  int amount_jpy;
  union { null, string } coupon_code = null;
}
```

これは通常、後方互換です。`coupon_code` を期待する新しい利用側でも、古いメッセージを読めます。欠けているフィールドはデフォルト値に解決されるからです。

互換でない例としては、移行境界を設けずに `amount_jpy` の型を `int` から `string` に変えるケースがあります。古いデータは数値としてエンコードされたままです。文字列を期待する新しい利用側は、そのフィールドを一貫して解釈できなくなります。

### 前方互換性

前方互換性とは、古い利用側が新しいスキーマで生成されたデータを読めることです。

同じ例を使うと、`order_id`、`customer_id`、`amount_jpy` しか知らない古い利用側は、たいてい新しく追加された `coupon_code` フィールドを無視できます。この変更は前方互換であることが多いです。

一方、フィールド名の変更は通常そうなりません。提供側が `amount_jpy` を `amount_minor` に変えても、古い利用側が依然として `amount_jpy` を期待しているなら、その間を自動的に橋渡しする仕組みはありません。フォーマットやツールがエイリアスをサポートし、しかも関係するすべてのコンポーネントがそれを正しく使っている場合は別ですが、それは簡単ではありません。

### 完全互換性

完全互換性は、隣接するバージョン間で両方向の安全性が保たれることを意味します。実際には、破壊的な変更よりも、追加的で、デフォルトを持ち、明示的なエイリアスで扱える変更が優先されます。

ここで重要なのは、「人間には安全そうに見える」変更が本当に安全とは限らないことです。フィールド名の変更は、値そのものが変わっていないので無害に見えがちです。しかし分散システムにとって、rename は見た目の変更ではありません。フィールドの見つけ方とデコードの仕方を変えてしまいます。

互換性が重要なのは、デプロイの現実があるからです。提供側と利用側は別々のタイミングで変わります。互換性ルールは、その時間差を生き延びるための仕組みです。

## 変更パターン

チームはしばしば互換性ルールをチェックリストとして覚えますが、そのルールが表しているのはシステム進化の非対称性です。

慎重に扱えば互換であることが多い変更には、次のようなものがあります。

- 任意フィールドの追加
- デフォルト値を持つフィールドの追加
- 古い利用側が無視できるデータでメッセージを拡張すること
- フォーマットがサポートし、利用側ロジックも値を解釈し続けられる範囲で、特定の数値型をより広い型へ拡張すること

一方、互換でないことが多い変更は次のとおりです。

- エイリアス対応や移行処理なしでフィールド名を変更すること
- 必須フィールドの論理型を別の型に変えること
- 任意フィールドを、バックフィル計画なしに必須へ変えること
- 同じ enum 値名を別の意味で再利用すること
- フィールド名や契約記述を変えずに単位だけを変えること

最後の例は、構文上の互換性だけでは誤解を招くことをよく示しています。`latency` が以前はミリ秒で、今は秒になったとします。スキーマはそのままパースできるかもしれません。それでも契約は壊れています。

## フォーマット比較

[Avro](https://avro.apache.org/)、[JSON Schema](https://json-schema.org/)、[Protobuf](https://protobuf.dev/) は、いずれも構造を記述するため、まとめて語られがちです。しかし運用上は、型付け、エンコーディング、進化の扱いに関して異なるトレードオフを持っています。

### Avro

Avro は、データパイプラインやイベントストリームに強く寄った設計です。スキーマが中心にあり、互換性ルールも十分に明示されているため、レジストリで予測しやすく強制できます。

Avro のレコードはタグベースではなくフィールドベースです。そのため、スキーマ進化ではフィールド名とデフォルト値が重要になります。

```json
{
  "type": "record",
  "name": "OrderCreated",
  "fields": [
    { "name": "order_id", "type": "string" },
    { "name": "customer_id", "type": "string" },
    { "name": "amount_jpy", "type": "int" },
    {
      "name": "coupon_code",
      "type": ["null", "string"],
      "default": null
    }
  ]
}
```

運用上の強み:

- デフォルト値と union によって追加的な進化を扱いやすい
- レジストリのツールが後方互換・前方互換チェックを備えていることが多い
- コンパクトなバイナリエンコーディングがストリーミングシステムと相性がよい

運用上の制約:

- フィールド名やデフォルト値が移行の成否を大きく左右するため、スキーマ設計の規律が重要になる
- rename のように小さく見える変更でも、軽いリファクタリングでは済まず、意図的な移行設計が必要になる

### JSON Schema

JSON Schema は柔軟で、API や検証境界で広く使われています。制約表現には非常に優れていますが、その柔軟さゆえに、進化ルールはツールごとにそろいにくいという面もあります。

```json
{
  "type": "object",
  "properties": {
    "order_id": { "type": "string" },
    "customer_id": { "type": "string" },
    "amount_jpy": { "type": "integer", "minimum": 0 },
    "coupon_code": { "type": ["string", "null"] }
  },
  "required": ["order_id", "customer_id", "amount_jpy"],
  "additionalProperties": false
}
```

運用上の強み:

- オブジェクト、パターン、範囲、ネスト構造に対して表現力の高い検証ルールを書ける
- HTTP ペイロード検証や JSON ネイティブなシステムに自然に適合する
- 人間にとっても読みやすく、意図を追いやすい

運用上の制約:

- Avro ほど互換性ポリシーが内在しておらず、何を安全とみなすかをチーム側で定義する必要がある
- `additionalProperties`、デフォルト値、緩い型付けの扱い方によって、許容度が大きく変わる
- バイナリ転送やコンパクトな進化を主眼に置いた設計ではない

JSON Schema は、統一された進化戦略そのものというより、強力な検証言語として理解する方が実態に合います。

### Protobuf

Protobuf は、サービス間通信とバイナリ効率のために最適化されています。進化の仕組みも、フィールドが名前だけでなく数値タグで識別される点で異なります。

```proto
syntax = "proto3";

message OrderCreated {
  string order_id = 1;
  string customer_id = 2;
  int32 amount_jpy = 3;
  string coupon_code = 4;
}
```

運用上の強み:

- コンパクトなバイナリエンコーディングと生成クライアントがサービス境界と相性がよい
- 多くのケースで、古い読み手は未知のフィールドを無視できる
- 表示名が変わっても、数値タグが安定した識別子として機能する

運用上の制約:

- タグの再利用は危険で、一度公開したフィールド番号は長期契約の一部になると考えるべき
- presence、デフォルト値、optional の意味は、バージョンや言語ランタイムによって繊細な差がある
- デコードが成功するため、意味が変わっていても安全だと誤解しやすい

Protobuf は、安全な追加的進化を比較的扱いやすい一方で、安易なフィールド廃止やタグ再利用には厳しいフォーマットです。

## スキーマレジストリ

イベントが共有ストリームを流れるようになると、互換性ルールをどこかに置かなければなりません。そこにスキーマレジストリの意味があります。

たとえば `orders.created` という Kafka トピックを考えます。提供サービスがメッセージを書き込み、不正検知サービスがリアルタイムに消費し、ストレージ用シンクがオブジェクトストレージへ保存し、分析パイプラインが後からその保存済みデータを読みます。レジストリがなければ、各コンポーネントがそれぞれ独自のスキーマ定義、独自のパース前提、独自のリリースタイミングを持つことになります。

レジストリは、いくつかの運用機能を集約します。

- スキーマバージョンの保管
- 登録時の互換性検証
- シリアライズとデシリアライズ時の識別子ベースのスキーマ解決
- どの subject や topic がどの契約を使っているかの可視化

実際には、この層をゼロから自作するチームは多くありません。通常は、イベント基盤、シリアライザスタック、クラウド環境とすでに統合されたツールを採用します。代表例としては、Kafka と Avro 中心の運用で広く使われる [Confluent Schema Registry](https://docs.confluent.io/platform/current/schema-registry/index.html)、複数のスキーマや API アーティファクトを扱える汎用寄りの [Apicurio Registry](https://www.apicur.io/registry/)、Kafka 環境で Schema Registry 互換の選択肢として使われることが多い [Karapace](https://karapace.io/)、AWS のマネージドなイベント基盤と連携する [AWS Glue Schema Registry](https://docs.aws.amazon.com/glue/latest/dg/schema-registry.html) などがあります。

これらのツールはパッケージングや前提エコシステム、扱えるアーティファクトの範囲は違いますが、解いている運用上の問題は同じです。悪いメッセージが稼働中のシステムへ広がる前に、スキーマ互換性の検査を済ませることです。

重要なのは、中央集約そのものではありません。ホットパスに届く前に悪い変更を落とせることです。

たとえば、`orders.created` の現行スキーマが `amount_jpy` を必須としているのに、ある提供側が互換性ルール上許されないまま `amount_minor` へ rename した新バージョンを登録しようとしたら、レジストリは登録を失敗させられます。その失敗は、デプロイ後に利用側が壊れてから気付くよりはるかに安い失敗です。

ストリーミングシステムでこの差は大きいです。メッセージは行儀よく待ってくれません。一度公開されると、状態ストア、コンパクト化トピック、デッドレターキュー、長期保持ストレージへすぐ広がります。互換でないスキーマを事前に弾くことは、もっとも低コストな制御点であることが多いのです。

レジストリは契約のスコープも可視化します。ひとつの内部トピックに付いたスキーマと、多数のチームが利用する公開イベントストリームに付いたスキーマは同じではありません。後者の方がデプロイ調整しにくいため、より厳しい互換性設定が必要になるかもしれません。

## 移行時の失敗

初期スキーマが完璧であることはほとんどありません。しかし、本番で痛みを生むのは、たいてい最初の定義よりも、その後の変更の仕方です。

### フィールド名の変更

rename は、ソース管理上では無害に見えるため、もっとも起きやすい失敗のひとつです。

提供側から見ると、`customer_id` を `account_id` に変えるのは、より適切なドメインモデルへの改善かもしれません。しかし既存の利用側から見ると、それは依存しているフィールドの削除であり、見慣れない新フィールドの追加です。一定期間両方の名前を併存させるか、フォーマットがエイリアスを提供し、しかもすべての利用側が実際にそれを使っているのでなければ、これは破壊的変更です。

### enum の拡張

enum 値の追加は追加的な変更に見えますが、下流ロジックはしばしば enum の値域を閉じた集合として扱います。

たとえば、注文ステータスの enum が最初は `pending`、`paid`、`cancelled` だけだったとします。あとから提供側が `refunded` を追加します。

```proto
enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  PENDING = 1;
  PAID = 2;
  CANCELLED = 3;
  REFUNDED = 4;
}
```

パース自体は成功するかもしれません。しかし、利用側の業務ロジックはそうとは限りません。ダッシュボードは新しい状態を落とし、アラートルールは誤分類し、状態遷移の実装は本来有効なトラフィックを default 分岐へ落としてしまうかもしれません。

これは、ワイヤ互換性と振る舞い上の互換性が違うことを示すよい例です。

### 必須化

任意フィールドを必須フィールドへ変えるのは、整理整頓に見えて実際には契約違反であることが多い変更です。

古い提供側が `coupon_code` を省略しているなら、それを必須とみなす新しい利用側は、過去メッセージも、追随が遅れている提供側のメッセージも安全には処理できません。リクエストレスポンス型のシステムであっても、ローリングデプロイ中はしばらく古い形のリクエストが流れます。

安全な手順は通常、段階的です。

1. まずフィールドを任意で追加する。
2. バックフィルや実装更新で一貫して値が入るようにする。
3. すべての提供側で値が入っていることを確認する。
4. そのあとで、全参加者が新しい要件を満たせると確認できた段階で検証を厳しくする。

### 意味のドリフト

もっとも高くつく失敗は、意味のドリフトです。スキーマ検証器では十分に捕まえられないからです。

たとえば `amount_jpy` が最初は税込み総額を意味していたのに、新しい経理プロセスに合わせて税抜き金額へ変えたとします。ただし、パーサを壊したくないのでフィールド名は変えません。

この場合も、すべてはデシリアライズできます。構文上は何も壊れていません。それでも売上ダッシュボード、不正検知のしきい値、返金ロジック、異常検知は、以前とは別の意味の値で動き始めます。

だから契約は型レベルの互換性だけでは足りません。フィールド名、説明、単位、業務上の定義までが、利用側が依存している契約の一部です。

## 意味の互換性

スキーマツールは構造チェックには強い一方で、意味のチェックには弱いです。

次のようなペイロードは、構造上は互換でも、運用上は互換ではありません。

- 通貨が円からドルへ変わっているのに整数型は同じまま
- `created_at` がイベント時刻から取り込み時刻へ変わっているのにタイムスタンプ型は同じまま
- `customer_id` がグローバル ID からテナント内ローカル ID へ変わっているのに文字列型は同じまま

こうした失敗は移行時に起きがちです。チームが「パーサが通るか」に意識を寄せすぎるからです。パーサが通ることは必要条件ですが、十分条件ではありません。

そのため、実用的な契約には、機械で検査できる構造と、人が管理する意味上の保証の両方が必要です。実務では、説明、単位、所有者、移行メモをスキーマの近くに置き、切り離された散文にしないことが重要になります。また、シリアライザの挙動だけでなく、利用側の振る舞いを念頭に置いて変更レビューする必要があります。

レビュー時に有効な問いは単純です。変更していない利用側がこの新しいメッセージを静かに受け入れたとして、それでも誤った答えを返し得るか。答えが yes なら、その変更は構文上は互換でも、意味上は安全ではありません。

## 契約の先

この段階で、メタデータはすでに運用の一部になっています。

イベントを公開してよいか、利用側がデコードできるか、検証器が不正ペイロードを拒否するか、ローリングデプロイを安全に進められるかを左右するからです。これは、第1回の列名やフィールド説明の世界から見ると大きな変化です。

ただし、それで全体像が完成するわけではありません。

契約が守るのは、提供者と利用者のあいだの論理解釈です。大規模な分析用ストレージへデータが保存されたあと、メタデータがスキャンコスト、predicate pushdown、エンコーディング選択、row group のレイアウト、ファイルレベルの計画にどう影響するかまでは説明しません。ストリーム契約は `created_at` がタイムスタンプであることは伝えられますが、Parquet ファイルを読む前にどれだけの row group を飛ばせるかまでは伝えられません。

次の境界はそこです。メタデータは、アプリケーション間のインターフェースであるだけでなく、計算効率そのものを左右し始めます。

第3回では、その段差へ進みます。データが大きな列指向ファイルに保存されるようになると、メタデータは提供者と利用者の論理契約であるだけでは足りません。エンジンがどの作業を実際に行うかを直接決めるようになります。
