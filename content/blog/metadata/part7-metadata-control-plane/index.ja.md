第6回は、メタデータが AI システムにとって意味インフラになる地点で終わりました。

プラットフォームが、鮮度、データ系統、ポリシー、モデル互換性、業務意味に関する機械可読な文脈を必要とするようになると、メタデータを受動的な文書とみなす考え方はすでに破綻しています。残る問いはさらに大きなものです。オーケストレーション、可観測性、統制に近い制御、再現性、自動 remediation が、実行時にメタデータへ依存するとき、メタデータはどんな役割を担うのか。

そこで、メタデータはコントロールプレーンになります。

本記事は [AI システムで必要な意味メタデータ](/ja/blog/metadata/part6-ai-semantic-metadata/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの実際の問いは、現代のプラットフォームが大量のメタデータを持っているかどうかではありません。それは持っています。問題は、どのメタデータがシステム挙動を直接制御し、それがデータ基盤と AI 基盤をどうまたいで調停し、メタデータを「表示するだけ」ではなく「作用させる」ようになったとき、どんなトレードオフが現れるかです。

## フィールド意味から協調へ

このシリーズは、曖昧なフィールドという狭い問題から始まりました。

列名、型、コメントがあるだけでも、人間とプログラムが同じデータセットを解釈するときのミスは減らせます。そこから範囲は広がりました。スキーマは提供者と利用者のあいだの契約になり、Parquet のメタデータは物理実行効率の一部になり、レイクハウスのメタデータは多数ファイルと並行ライターをまたぐテーブル状態の権威ある記録になり、クエリプランナは統計、catalog、inventory に依存し、AI システムは特徴量、チャンク、埋め込み、出所、ポリシーを解釈するための意味メタデータを必要とするようになりました。

各段階で、メタデータはより運用制御に近づきました。

この流れが重要なのは、現実のプラットフォームでは、これらが別々のカテゴリとして分離していないからです。現代の基盤は、互換性のためのスキーマメタデータ、計画のためのファイル・snapshot メタデータ、影響分析のための系統情報、アクセス判断のためのポリシーメタデータ、検索や特徴量配信のための意味メタデータを同時に使います。そうした層がすべて active になると、メタデータはもはや sidecar ではありません。何が実行できるか、何が有効か、何が見えるか、何を更新すべきか、何を止めるべきかを、プラットフォーム全体へ伝える層になります。

それをコントロールプレーンと呼ぶのは比喩ではありません。意思決定がどこで行われているかを説明する、工学的な表現です。

## ここでいうコントロールプレーンとは何か

分散システムでは、control plane は desired state、policy、topology、coordination logic を管理し、data plane は実際の作業を行います。[Kubernetes](https://kubernetes.io/) は分かりやすい例です。API server、scheduler、controller が「何をどこで動かすか」を決め、コンテナは data plane で実行されます。

同じ分離は、データ基盤や AI 基盤にもあります。アーキテクチャがそれほど明示的でない場合でもです。

data plane には、クエリエンジン、ストリーム処理、オブジェクトストレージ読み取り、バッチジョブ、モデル推論サービス、ベクトル検索、タスクワーカーなどが含まれます。これらは、ファイルを読み、レコードを処理し、埋め込みを計算し、モデルを学習し、リクエストへ答える側です。

control plane は、次のようなことを決める層です。

- どのテーブル snapshot が現行か
- スキーマ変更は互換か
- どの partition や file が実行対象として適格か
- どの asset に機微データが含まれ、masking や deny が必要か
- どの上流変更が再計算や再インデックスを引き起こすべきか
- ある特徴量がオンライン配信に使えるだけの鮮度を持つか
- 取得されたチャンクが認可済みで、現行で、意味的に比較可能か
- 失敗したパイプラインを retry すべきか、pause すべきか、escalate すべきか

これらの判断は、ほとんどがメタデータ解釈の問題です。

SQL、workflow 定義、feature code、retrieval prompt は意図を表します。しかしプラットフォームは、意図だけでは安全に実行できません。オブジェクト、依存関係、バージョン、ポリシー、品質シグナル、所有境界の機械可読な記述が必要です。その記述層こそが、協調を可能にします。

したがって、メタデータをコントロールプレーンと呼ぶとは、具体的には次の3点を意味します。

第一に、メタデータが許容される状態空間を定義することです。プラットフォームは、どんなオブジェクトが存在し、どの版が権威あるか、どんな操作が許されるかを知ります。

第二に、メタデータが高コストな実行前の意思決定ロジックを駆動することです。プランナ、オーケストレータ、ポリシーエンジン、自動 remediation システムが、行動経路を選ぶ前にメタデータを参照します。

第三に、メタデータが実行後のフィードバックループを閉じることです。可観測性イベント、系統レコード、品質チェック、run 結果がメタデータを更新し、それが次の意思決定を変えます。

最後のループが、静的カタログを運用システムへ変えます。

## オーケストレーションはメタデータに依存します

ワークフローオーケストレーションは、メタデータがすでにコントロールプレーンとして振る舞っていることを示す、もっとも分かりやすい場所のひとつです。

[Apache Airflow](https://airflow.apache.org/)、[Dagster](https://dagster.io/)、[Prefect](https://www.prefect.io/)、[Argo Workflows](https://argoproj.github.io/workflows/) のようなシステムは、タスク、依存関係、パラメータ、run、状態遷移に関するメタデータを基に作業をスケジュールします。そのメタデータの一部はコード内にあり、一部は backing database にあり、一部は実行中に動的に出てきます。いずれにしても、オーケストレータは「作業の記述」を通じて作業を調停しています。

日次の変換パイプラインを考えてみます。

実際の計算タスクは Spark、dbt、SQL warehouse、コンテナ化ジョブで走るかもしれません。しかし、その計算を始める前に、オーケストレータは次を知る必要があります。上流 asset は完了しているか、どの partition window が対象か、どのコード版を使うべきか、どの retry policy が適用されるか、どの secret や service account が許されるか、下流データセットは incremental に materialize すべきかフル再構築すべきか。これらは raw data-plane の問題ではありません。メタデータに基づく制御判断です。

[Dagster の asset model](https://docs.dagster.io/guides/build/assets) は、asset、lineage、partition、freshness 期待、software-defined asset を明示する点で、よい例です。scheduler は shell command の連なりではなく、asset について推論できます。[Airflow datasets](https://airflow.apache.org/docs/apache-airflow/stable/authoring-and-scheduling/datasets.html) も、固定時刻だけではなく、データ到着を基準に下流スケジューリングできる方向へ進んでいます。

メタデータが豊かになるほど、オーケストレーションが信頼しやすくなるのはこのためです。

もしパイプラインが「Job B は Job A のあと」ということしか知らなければ、順序は守れます。しかし partition key、上流 version、期待行数範囲、所有者、SLA、データ品質アサーションまで知っていれば、もっと強い判断ができます。不要な作業をスキップし、安全でない作業を遅らせ、特定 partition だけを backfill し、インシデントを適切なチームへ振り分けることができます。オーケストレータは単なる cron の置き換えではなく、メタデータ解釈器になります。

同じパターンはストリーミングにもあります。[Confluent Schema Registry](https://docs.confluent.io/platform/current/schema-registry/index.html) や [Apicurio Registry](https://www.apicur.io/registry/) のような schema registry は、Kafka topic 上のバイトを運ぶわけではありません。しかし、提供者と利用者が安全に進化できるかを直接制御します。互換性メタデータが、トラフィックを通すか fail fast させるかを決めます。これは broker のコア data path の外にあっても、明らかに control-plane 的な振る舞いです。

## ポリシー、可観測性、再現性

control plane はスケジューリングだけではありません。ガードレールと証跡も含みます。

現代のプラットフォームでは、ポリシー適用がメタデータに強く依存します。単純なテーブル単位権限では足りず、タグ、機密度ラベル、法域、データ系統、想定利用、保持ポリシー、契約スコープ、モデルリスク分類に応じた判断が必要だからです。そうした判断は、メタデータが機械評価できる形で構造化されているときに初めて現実的になります。

[Open Policy Agent](https://www.openpolicyagent.org/) は一般的なパターンをよく示しています。ポリシーは、リソースとアイデンティティに関する事実を評価して、何が許されるかを決めます。データプラットフォームでは、その事実は catalog、タグ、系統グラフから来ることが多いです。PII ラベル付き列だから masking を適用する、承認済みのデータ系統がない特徴量だから model serving が拒否する、メタデータ上 internal-only とされたチャンクだから retrieval service が落とす。こうしたことはすべてメタデータの評価です。

可観測性も、別の話のようで実は同じ依存を持ちます。[Monte Carlo](https://www.montecarlodata.com/)、[Bigeye](https://www.bigeye.com/)、あるいは [OpenMetadata](https://open-metadata.org/) や [DataHub](https://datahub.com/) を軸にしたオープンな仕組みは、インシデント、鮮度チェック、スキーマ変化、アサーション、所有者情報をデータ asset に結び付けることで機能します。メタデータがなければ、異常はログ中の数値にすぎません。メタデータがあれば、どのテーブルが変わり、誰が所有し、どの下流ジョブが依存し、どの程度深刻かまで分かります。

だから、可観測性はテレメトリ問題であると同時に、メタデータ問題でもあります。

メトリクス、トレース、ログは事象を記録します。メタデータは、その事象が運用上何を意味するかを与えます。deprecated な内部テーブルでの null 値増加と、本番の信用リスクモデルに流れる規制対象データソースでの null 値増加は、同じ扱いではありません。自動で対応するには、所有者メタデータ、依存関係メタデータ、品質ルール、ポリシー文脈が必要です。

再現性も同じ論理です。クエリ結果、モデル予測、検索回答を再現するには、コードだけでは足りません。バージョン、入力、環境、データ系統に関するメタデータが必要です。

分析なら、テーブル snapshot、スキーマ version、変換コードの revision かもしれません。機械学習なら、特徴量定義、学習データセット、ハイパーパラメータ、モデル artifact version、environment hash であり、[MLflow](https://mlflow.org/) のようなシステムがこの構造をよく示します。検索系なら、文書 version、チャンクメタデータ、埋め込みモデル version、index build ID、リクエスト時に適用したポリシーフィルタかもしれません。

こうしたメタデータが欠けていれば、プラットフォームは実行はできても、何が起きたかを信頼して説明したり再現したりはできません。それは文書不足ではなく、制御の失敗です。

## アクティブメタデータはシステム挙動を変えます

コントロールプレーンの議論がもっとも強くなるのは、メタデータが人間に情報を見せるだけでなく、システムの動作を直接起こすようになったときです。

いわゆる active metadata です。用語としては曖昧に使われがちですが、工学的なパターンは具体的です。メタデータの変化、あるいはメタデータから導かれたイベントが、プラットフォーム内の別コンポーネントで自動挙動を起こします。

よくある例は、すでに日常的です。

- スキーマ変更イベントが互換性チェックを起動し、下流が壊れる変更ならデプロイを止める
- 系統グラフがソーステーブル変更の影響先ダッシュボードや特徴量ジョブを特定し、インシデントを起票したり下流パイプラインを止めたりする
- データ品質アサーションの失敗が asset を unhealthy とみなし、下流公開を防ぐ
- テーブル snapshot の更新が、新しく追加された partition だけの増分処理を起動する
- 文書更新が、全文書の再構築ではなく、影響チャンクだけの再 chunk 化と再埋め込みを起動する
- 機密タグが、クエリ結果の自動 masking や AI ツールからの利用拒否を引き起こす

どの例でも、プラットフォームは raw data value だけを見て動いていません。メタデータの関係性と状態遷移を見て動いています。

ここで、catalog、lineage system、orchestration system の境界は重なり始めます。[OpenLineage](https://openlineage.io/) は run や依存関係のメタデータを出し、[Marquez](https://marquezproject.ai/) はそれを蓄積し、[OpenMetadata](https://open-metadata.org/) や [DataHub](https://datahub.com/) のようなメタデータプラットフォームは、lineage、ownership、tag、assertion、action を結び付けます。その状態を受けて、orchestrator が run、retry、notify、quarantine を決められるようになります。

active metadata かどうかを見極める実用的な問いは単純です。メタデータが変わったら、システム挙動は自動的に変わるか。yes なら、そのメタデータはすでに運用インフラです。

完全自律である必要はありません。人間承認を挟む remediation flow でも同じです。たとえば、プラットフォームはメタデータを使ってスキーマ障害の blast radius を絞り、影響 asset 一覧を作り、backfill plan を提案し、適切な owner に承認を回せます。人間は loop の中に残りますが、制御ロジックはメタデータ駆動です。

## データ基盤と AI 基盤の制御面は収束しています

control-plane 層では、データ基盤と AI 基盤を別物として考えることの有効性が下がっています。

ストレージや実行エンジンは確かに違います。warehouse、lakehouse、特徴量ストア、vector index、model registry、online inference service は、それぞれ別の data plane を持ちます。しかし control plane の概念は似通っています。lineage、ownership、freshness、policy scope、versioning、compatibility、semantic meaning、observability state です。

企業データと結び付いた retrieval-augmented generation を考えると、この収束は見えやすいです。

ソースはデータプラットフォームが管理するテーブルや文書かもしれません。それらは変換され、チャンク化され、埋め込み化され、インデックスされ、フィルタされ、最終的にアプリケーションや agent に検索されます。その過程で、プラットフォームは source から chunk への系統情報、source から retrieval object へのポリシーラベル、公開ワークフローから index rebuild への鮮度期待、業務概念から検索可能 asset への意味マッピングを保持しなければなりません。

これはデータプラットフォームと別のものではありません。その延長です。

特徴量プラットフォームでも同じことが逆方向から見えます。特徴量ストアは、データスタックから batch lineage や warehouse semantics を受け継ぎつつ、online serving の鮮度、point-in-time の正しさ、モデル互換性、推論時ポリシーチェックといった AI 寄りの制御も必要とします。制御ロジックは両領域をまたぎます。

そのため、catalog、orchestration、lineage、observability、semantic layer、model registry を孤立した製品カテゴリとして扱うより、それらを接続する方向へ進むチームが増えています。有用な抽象は「catalog 対 ML ツール」ではなく、「メタデータがスタック横断で十分強く流れ、機械が安全に協調できるか」です。

その接続ができると、同じメタデータグラフが複数の判断を支えます。

- プランナがファイルと join を選ぶ
- オーケストレータが次に materialize すべきものを決める
- ポリシーエンジンが、agent がアクセスしてよい asset を絞る
- 可観測性システムが、下流 blast radius に基づいてインシデント優先度を付ける
- 再現性ワークフローが、予測や回答の正確な入力を復元する

消費するシステムは違っても、制御面は共有されています。

## 現実のプラットフォームにあるトレードオフ

メタデータをコントロールプレーンとして扱うのは有用ですが、設計上の圧力も生みます。

第一のトレードオフは、中央集約と federated ownership のあいだです。

強く中央集約されたメタデータシステムは、より強いグローバル保証を与えやすくなります。タグ、lineage、policy、quality status を一箇所で統合できます。しかし、すべてのドメインチームが platform team のモデリングや asset onboarding を待たねばならないなら、ボトルネックにもなります。federated ownership はローカル精度と浸透を高めますが、メタデータが相互運用可能であるための共通構造を platform 側が十分に強制できる場合に限ります。

第二は、柔軟性と保証のトレードオフです。

チームは、新しいデータセット、特徴量、プロンプト、検索ソースを素早く公開したいと思います。強いメタデータ要件は摩擦に見えることがあります。しかし、弱いメタデータは弱い保証しか生みません。ownership、lineage、versioning、freshness、policy tag が任意なら、プラットフォームは安全に自動化できません。実際に成功しているプラットフォームは、運用安全のために必須なメタデータと、より豊かな discoverability のための任意メタデータを分けています。

第三は、自動化と信頼のトレードオフです。

自動 quarantine、masking、backfill、schema blocking、選択的再インデックスは、運用負荷を大きく減らせます。しかし、その自動アクションを支えるメタデータが正しいと、運用者が信頼できる必要があります。lineage が不完全だったり、tag が stale だったりすると、自動アクションは誤った asset を隠したり、本当の blast radius を見落としたりします。だから、自動制御面には、観測可能な信頼境界、承認経路、巻き戻し可能性が必要です。

もうひとつのアーキテクチャ上の圧力は、canonical truth と derived cache のあいだにあります。論理的にはひとつのメタデータグラフが source of truth でも、プランナ、インデックス、policy engine はそれぞれ自分向けに最適化された view を必要とします。すると、よくある分散システム問題が再び現れます。同期です。あるメタデータ利用者が stale state を見れば、中央カタログが正しくても、誤った制御判断を下す可能性があります。

こうしたトレードオフは、control-plane view を否定するのではありません。構築が難しい理由を説明します。

## 機械が操作するシステムでも、土台はメタデータです

agentic system はコントロールプレーンの問題を見えやすくしますが、本質を変えるわけではありません。

タスクを計画し、ツールを選び、コンテキストを取得し、ワークフローを起動するエージェントは、free text だけを頼りに動くべきではありません。どんなツールがあり、どんな入力を受け、どんな asset が認可され、どれだけの鮮度が必要で、どんな policy が適用され、結果をどう追跡すべきかについての、機械可読な制約が必要です。つまり、安全に行動を制約できるほど豊かなメタデータが必要です。

この構造は、ツールスキーマ、アクセス範囲、retrieval filter、run lineage、approval checkpoint を持つ本番システムにすでに現れています。大規模モデルを使っているから信頼できるのではありません。モデルが何を要求でき、ランタイムが何を実行でき、結果をあとからどう監査できるかを、メタデータが十分に制約しているときにだけ、より信頼できます。

機械が操作するインフラを grounded に捉えるとは、そういうことです。

重要な変化は、機械がエンジニアを置き換えることではありません。プラットフォーム挙動のより多くが、機械が直接評価できるメタデータによって仲介されるようになっていることです。スキーマを設計し、policy をレビューし、ownership を定義し、refresh rule を決め、ドメインをモデル化し、どこまで自動化するかを決めるのは、依然としてエンジニアです。しかし、その判断が構造化メタデータとして表現された瞬間、プラットフォームは人間が手で維持できない規模でも一貫してそれを実行できます。

これが、このシリーズの終点です。

メタデータは、データの周縁にある説明として始まりました。その後、互換性ロジック、物理計画入力、テーブル協調状態、AI システムの意味コンテキスト、そして最後にはオーケストレーション、統制に近い制御、可観測性、再現性、自動ワークフローが依存する機械可読レイヤーになりました。この変化は、現代のデータプラットフォームと AI プラットフォームにすでに表れています。

メタデータは、受動的な説明からアクティブな機械協調へ進化しました。実務の工学として言えば、それがメタデータがコントロールプレーンになるということです。
