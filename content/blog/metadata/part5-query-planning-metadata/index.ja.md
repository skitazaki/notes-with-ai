第4回は、レイクハウスのテーブルが権威あるメタデータ構造になった地点で終わりました。

エンジンは、データセットを緩いファイル集合として扱いません。まずテーブルバージョンを解決し、どのファイルが live かを知り、そのあとでどんな作業をスケジュールするかを決めます。この変化が重要なのは、スキャン演算子がバイトをデコードし始めるより前の段階で、クエリ実行の質がほぼ決まるからです。

そこで、メタデータはクエリ計画になります。

本記事は [レイクハウス規模のメタデータ](/ja/blog/metadata/part4-lakehouse-metadata/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの実際の問いは、オプティマイザが抽象的に賢いかどうかではありません。実行が始まる前にどんな情報を必要とし、その情報がどう計画を変え、メタデータが弱い、古い、誤解を招くときに何が起きるのかです。

## 計画前のメタデータ

単純な分析クエリを考えてみます。

```sql
SELECT customer_id, sum(amount_jpy) AS revenue
FROM sales
WHERE order_date >= DATE '2026-05-01'
  AND order_date < DATE '2026-05-08'
  AND region = 'JP'
GROUP BY customer_id;
```

エンジンは、`sales/` 配下のすべての Parquet ファイルを開くところから始めません。

まず、次のような計画上の問いに答えます。

- 現在のカタログにおいて `sales` はどのテーブルを指すか
- このクエリから見えるのは、どのスナップショットあるいはテーブルバージョンか
- そのバージョンにはどのファイルが属するか
- テーブルはどうパーティション分割されているか
- テーブル、パーティション、ファイル、row-group のどの粒度で統計があるか
- 投影とフィルタに必要な列はどれか
- より大きいクエリの中で、結果は broadcast join できるほど小さいか
- 計画を簡略化できる制約やキー関係は分かっているか

これらはすべてメタデータの問いです。

だから、クエリ計画はメタデータ解釈フェーズだと捉えるべきです。SQL テキストは意図を表しますが、その意図のコストがどれくらいか、どのオブジェクトに対して適用されるか、データセットのどこを安全に無視できるかをプランナに伝えるのはメタデータです。オプティマイザは、まだ生のバイトについて考えているのではありません。バイト、ファイル、パーティション、スキーマ、関係性の「記述」について考えています。

その記述層は、複数の場所に同時に存在します。カタログエントリ、テーブルフォーマットのメタデータ、パーティションマップ、file statistic、システムによっては列レベルの要約や制約です。表に出る形はエンジンごとに違いますが、[DuckDB](https://duckdb.org/)、[Apache Spark](https://spark.apache.org/)、[Trino](https://trino.io/) に共通するパターンは同じです。計算品質は、メタデータ品質に強く依存します。

## プランナへの入力

プランナが使うメタデータ入力は、どんな判断を支えるかで分けて考えると分かりやすくなります。

もっとも分かりやすいのはスキーマです。スキーマメタデータがなければ、式の型も、predicate の合法性も、各ファイルフォーマットに必要な物理リーダーも分かりません。スキーマは、型変換、null 許容の推論、意味を壊さずにフィルタをスキャンへ押し込めるかどうかも左右します。

2つ目の大きな入力は統計情報です。行数、distinct 件数、null 比率、min/max、ヒストグラムに近い要約、ファイルサイズなどは、カーディナリティやコストの見積もりに使われます。プランナにとって完全な真実は不要ですが、複数の選択肢を比較できるだけの強いシグナルは必要です。

パーティションマップ と file inventory は別の問いに答えます。該当データは物理的にどこにあり得るか、という問いです。レイクハウステーブルなら、その答えは Iceberg のマニフェスト、Delta のトランザクションログ状態、あるいはカタログに裏付けられた file listing 抽象から来るかもしれません。データウェアハウスなら内部ストレージメタデータかもしれません。いずれにせよ、プランナはスキャン作業を考える前に、候補集合を境界付ける必要があります。

制約や、データ系統に近いメタデータもさらに信号を追加します。主キーや一意性制約は、完全に強制されていなくても、join cardinality の推定を助けることがあります。鮮度メタデータ、スナップショット時刻、出所情報（provenance）を示すフィールドは、統制された環境では「どのテーブルバージョンを計画対象にしてよいか」にも影響します。これらは伝統的なオプティマイザ入力とは言い切れませんが、実行前に安全で妥当なデータを選ぶうえで、ますます重要になっています。

ここにシステム上の境界があります。

- スキーマメタデータは、データが構造的に何を意味するかを説明する
- ロケーションメタデータは、候補データがどこにあるかを説明する
- 統計メタデータは、条件に合うデータがどれくらい残るかを推定する
- 制約と出所情報のメタデータは、どの解釈が安全かを制限する

一般的なオプティマイザ図は、この区別を見えにくくします。パーサ、論理計画、物理計画の箱だけを並べがちです。しかし、その各変換は、テキストとしてのクエリを「境界づけられた実行問題」へ変えるメタデータに支えられています。

## プルーニング

プルーニングは、メタデータが本格的なデータスキャンより前に実行時挙動を変えることを、もっとも分かりやすく示す例です。

`sales` が `order_date` でパーティション分割されており、各パーティションが `region` と `order_date` の min/max 統計を持つ複数の Parquet ファイルで構成されているとします。上のクエリは 1 週間と 1 地域だけを参照しています。健全なプランナなら、テーブルの大部分を段階的に除外するはずです。

最初はパーティションプルーニングです。パーティションマップに 4 月、5 月、6 月の日次パーティションがあるなら、プランナは `2026-05-01` から `2026-05-07` までのパーティションだけを残せます。この判断は、ファイルフッターに触れず、テーブルメタデータだけから行えることが多いです。

次は file pruning です。選ばれたパーティションの中でも、ファイルレベルメタデータを見ると、あるファイルが `region = 'US'` しか含まない、あるいは遅延 compaction や混在したパーティション transform の結果として、対象外の date range しか持たないことがあります。そうしたファイルは、タスクスケジューリング前に候補から外せます。

最後に、残ったファイルの内部で row-group pruning が行われます。エンジンが surviving file の Parquet フッターを読むと、predicate を満たし得ない row group を飛ばせます。

重要なのは、プランナがより細かい粒度のメタデータを使って、候補集合を何段階にも絞ることです。

1. カタログとテーブルメタデータが、現在のテーブルバージョンを定義する
2. パーティションメタデータが、候補パーティションを制限する
3. ファイルメタデータが、候補ファイルを制限する
4. ファイルフォーマットの統計が、候補 row group を制限する

だから、ファイルやパーティションの除外は「あとで少し読み込みが減る」程度の話ではありません。そもそもどれだけの仕事をスケジュールするかを変えます。ファイルが減れば、オブジェクトストレージのリクエスト数も、タスク数も、解凍量も、メモリ圧力も、中間状態も減ります。

[Apache Spark SQL](https://spark.apache.org/sql/) では、この層構造が explain plan に現れやすく、パーティションフィルタ と pushed filter が別々に見えることがあります。[Trino](https://trino.io/) でも、コネクタごとの predicate pushdown や split generation を通じて似た挙動が見えます。[DuckDB](https://duckdb.org/docs/stable/data/parquet/overview.html) は、よりコンパクトに同じ考え方を示すことが多く、Parquet を直接問い合わせても、ファイルや row-group の統計を使って無関係なセグメントを避けられます。

エンジンによって境界は違います。DuckDB は組み込みでファイルを直接読むことが多いため、ファイルフォーマットに近い場所で計画します。Spark や Trino は、カタログ、テーブルフォーマット、分散 split スケジューラを通じて計画することが多いです。それでも共通する原理は同じです。プルーニングは、スキャン実行の前に存在するメタデータに依存します。

## 統計情報

プルーニングが減らすのはスキャン候補です。その後の計画の形を決めるのが統計情報です。

次の join クエリを考えます。

```sql
SELECT o.order_id, c.segment
FROM orders o
JOIN customers c
  ON o.customer_id = c.customer_id
WHERE o.order_date >= DATE '2026-05-01'
  AND c.country_code = 'JP';
```

プルーニングのあとでも、プランナは join 順序、join アルゴリズム、データ移動戦略を決めなければなりません。そのために必要なのが、カーディナリティ推定です。

メタデータが「`country_code = 'JP'` で絞った `customers` は 50,000 行しか残らないが、絞った `orders` は 2 億行残る」と示していれば、小さい側を broadcast する方が、両方を repartition するより安いかもしれません。逆に、絞っても顧客テーブルが大きく、しかも偏っていると分かれば、プランナは partitioned hash join や別の join 順序を選ぶかもしれません。

ここで、コストベース最適化は実際的なものになります。オプティマイザは魔法を使っているのではありません。メタデータから導いた推定行数、行幅、ファイルサイズ、演算子コストを基に、候補プランを比較しています。

特に重要な推定は、次の3つです。

- フィルタ後のカーディナリティ
- join predicate の選択性
- 投影後のデータサイズ

推定が悪いと、どれも歪みます。predicate が極端に選択的だと誤認すれば、実際には危険な broadcast plan を選ぶかもしれません。join key の distinct 値を過小評価すれば、中間結果を膨らませる join 順序を選ぶかもしれません。null 統計が弱ければ、predicate を簡約できず、aggregation コストも見誤ります。

ここで見落とされがちな限界があります。コストベース最適化の「コストベース」は、入力されるメタデータの質に依存します。統計がなければ、オプティマイザはヒューリスティックへ退避します。ヒューリスティックは時に妥当ですが、証拠ではありません。

この点では、エンジンごとの差も見えます。

Spark は、カタログやテーブルフォーマット統合を通じて収集したテーブル・列統計に依存することが多く、その後、adaptive execution によって実行時サイズが見えてから一部の判断を補正します。Trino はコネクタが提供する table statistic への依存が大きく、どれだけのメタデータを提示できるかが計画品質を左右します。DuckDB はローカルファイルメタデータから素早く有用な情報を引き出せますが、周辺のテーブルメタデータが明示されている方が、アドホックな再構築よりやはり強いです。

3者に共通するのは、統計情報が「完成した計画を微調整するもの」ではないということです。そもそも、どの計画を plausible とみなすかを決めています。

## カタログ

計画は統計より前から始まっています。名前解決の段階からです。

クエリが `catalog.sales.orders` を参照するとき、エンジンは単なるストレージパス以上のものを解決しなければなりません。カタログやメタデータサービスは、どのオブジェクトが存在し、どのスキーマバージョンが現行で、どのテーブルフォーマットが使われ、どのスナップショットが見え、より深い計画メタデータがどこにあるかをプランナへ伝えます。

この名前解決ステップは、SQL の見た目の軽さのせいで過小評価されがちですが、実際には planning path の一部です。

[Apache Iceberg](https://iceberg.apache.org/) のテーブルは分かりやすい例です。プランナは通常、カタログを通じてテーブルを解決し、テーブルメタデータファイルを読み、現在のスナップショットを見つけ、そのあとでマニフェスト pruning や file selection に入ります。[Delta Lake](https://delta.io/) も、トランザクションログ状態とチェックポイントを通じて同じ役割を果たします。metastore 的なシステムが入口になっていても、権威あるファイルセットはカタログレコードそのものではなく、テーブルフォーマットメタデータ側にあるかもしれません。

つまり、カタログは少なくとも次の4つの意味で計画に参加しています。

- 名前解決と権限確認
- スキーマとテーブルプロパティの取得
- スナップショットまたはバージョンの発見
- プルーニングやコスト推定に使う、より深いメタデータ構造へのアクセス

分散エンジンでは、カタログはレイテンシと障害境界も形作ります。メタデータ解決が遅ければ、すべてのクエリがその計画オーバーヘッドを払います。カタログ状態がリージョン間やコネクタ間で不整合なら、ユーザーには同じテーブルに見えても、エンジンごとに異なる候補ファイル集合へ到達するかもしれません。権限解決が誤っていれば、プランナは本来見えるはずの列やパーティションを最初から候補に入れられません。

この運用上の役割があるから、現代のメタデータサービスは飾りではありません。計画インフラそのものです。

## 失敗モード

メタデータ駆動の計画は、メタデータが最新で信頼できるときにうまく働きます。その前提が崩れると、急速に劣化します。

もっともよくあるのは stale statistic です。テーブルが直近で compaction されたり repartition されたり、特定の取り込みパターンで偏りが増えたりしているのに、プランナが先週の推定をまだ使っている状況です。クエリの論理的正しさは保たれても、選ばれるプランは必要以上に高価になります。

不完全なメタデータは、古いメタデータとは別問題です。そもそも十分な詳細を返さないテーブルもあります。コネクタが行数は返すが列統計を返さない、ファイル統計はあるがパーティションマップが弱い、カタログはパーティションパスを知っているがスナップショットに対する正確な live ファイルセットは持っていない。こうした状況では、オプティマイザは入力が足りないため、プラン比較がうまくできません。

欠落より悪いのは、誤解を招くメタデータです。壊れた min/max、楽観的すぎる null 比率、正しくない一意性前提があると、プランナは必要なデータを除外しそうになったり、病的な join 戦略を選んだりします。多くのクエリエンジンは、正しさを性能より厳しく守るので、メタデータを十分に信頼できないときは安易なプルーニングを避けます。それでもメタデータ品質が悪いと、広いスキャンと単純なヒューリスティックへ防御的に退避することになります。

ここでも、データ系統に近いメタデータは重要です。統計が現在のスナップショットに対応しているか、派生テーブルが上流ソースに対してどれくらい遅延しているかが分からなければ、鮮度が重要なワークロードに対してプランナは自信を持てません。普通のダッシュボードではレイテンシが悪化するだけかもしれませんが、プロダクト内データや自動判断系では「そのクエリ結果を使ってよかったのか」自体に関わります。

実務的な教訓は単純です。メタデータ品質は、クエリ性能エンジニアリングの一部です。CPU、メモリ、スキャン量は監視しても、統計メンテナンスやカタログ健全性は裏方の管理業務として扱われがちです。しかし、その切り分けは人工的です。planning metadata が弱ければ、実行品質も弱くなります。

## 適応の境界

メタデータ駆動の計画は強力ですが、それだけで全てではありません。多くのエンジンは、実行中に適応もします。

分かりやすい例は Spark の adaptive query execution です。初期計画は依然としてメタデータと推定に基づいて構築されますが、実際の shuffle パーティションサイズのような実行時情報によって、あとから計画が修正されることがあります。フィルタ後に初めて broadcast join が現実的になることもあります。偏ったパーティションは、推定ではなく実測サイズを見て分割されることもあります。

これは、メタデータの重要性を下げる話ではありません。境界を明確にする話です。

メタデータ駆動の計画が答えるのは、「実行前に分かっていることだけを使うと、何をスケジュールすべきか」です。実行時適応が答えるのは、「演算子が実際の測定値を出した今、どの部分の計画を修正すべきか」です。

両者は補完関係にあります。

- メタデータは、作業が始まる前に探索空間を減らす
- 実行時適応は、実測結果に基づいて初期判断を修復・微調整する

メタデータがなければ、エンジンは余計な仕事をスケジュールし、悪い前提から出発します。適応がなければ、ランタイムの証拠が初期推定と矛盾しても、悪い計画に縛られたままになります。成熟したシステムには両方が必要ですが、使うタイミングも証拠も違います。

この区別は、プランナの知性を過大評価しないためにも重要です。コストモデルがあるからといって、クエリエンジンが全知になるわけではありません。実行前に知っているのは、メタデータが教えてくれる範囲だけです。メタデータが弱ければ、初期計画は本質的に近似になります。

## AI ワークロード

従来のクエリ計画が主に気にするのは、型付き構造、ファイルレイアウト、統計要約です。AI ワークロードは、それらすべてを必要としたうえで、さらにメタデータの境界を押し広げます。

特徴量ストアのクエリは、依然としてパーティションプルーニングやコスト推定の恩恵を受けます。しかし検索パイプラインは、それに加えて意味メタデータを必要とします。埋め込みモデルのバージョン、チャンクの出所情報、文書タイムスタンプ、言語、アクセス方針、ソースの信頼性、タスク固有の関連度シグナルなどです。これらは後で画面表示するためのラベルではありません。何を検索できるか、何をフィルタできるか、何を結合・再ランク付けできるか、何を信頼してよいかを決めます。

だから、メタデータの境界は広がっています。関係データの計画では、メタデータは「どのファイルをスキャンするか」「join を broadcast すべきか」といった問いに答えます。AI ネイティブなシステムでは、それに加えて「このチャンクはどの文書バージョンに属するか」「2つのベクトルは互換なエンコーダから来ているか」「データ系統情報（lineage）上、このコンテキストは下流回答に使ってよいか」「鮮度ルール上、この取得結果は不適格か」といった問いにも答える必要があります。

計画のパターン自体は変わっていません。演算子が変わっただけです。

- スキーマと統計は、構造化テーブルを記述する
- 意味属性と出所情報は、検索対象オブジェクトを記述する
- カタログとレジストリは、現在の論理オブジェクト集合を解決する
- プランナは、それらの信号を使って高価な実行の前に作業範囲を絞る

したがって、メタデータの次の段階はクエリ計画を捨てることではありません。古典的なテーブルやファイル統計を超えて、意味、出所情報、データ系統、検索コンテキストを計画入力へ含めることです。

## メタデータ依存の計画

クエリエンジンにとってメタデータが重要なのは、計画が計算問題である前に情報問題だからです。

スキーマは、どんな操作が合法かをエンジンへ伝えます。パーティションマップ と file inventory は、一致するデータがどこにあり得るかを境界付けます。統計は、フィルタ後や join 後にどれくらいのデータが残るかを見積もります。カタログは、それらの判断が適用されるバージョン付きオブジェクトを解決します。こうしたメタデータ層が健全なら、プランナは積極的にプルーニングし、よりよい join 戦略を選び、少ない仕事だけをスケジュールできます。逆に、それらが弱い、古い、誤解を招くなら、コストベース最適化は当てずっぽうに近づき、実行時適応が悪い出発点からの回復を強いられます。

だから、メタデータは文書化層ではなく、実行計画層として扱うべきです。プランナは、ほとんどのデータバイトに触れる前に、まずメタデータを消費して物理戦略へ変換します。

第6回では、この考え方を AI システムへ広げます。ワークロードが検索、埋め込み、出所情報、意味フィルタに依存するようになると、型付きテーブルとファイル統計だけでは足りません。AI ネイティブなシステムには、意味、データの出所、データ系統、検索コンテキストを first-class の計画入力として表せるメタデータが必要になります。
