第2回は、スキーマが運用上の意味を持つ境界で終わりました。提供者と利用者が同じ構造に依存し始めると、メタデータはもはや説明だけのものではありません。変更を制約します。

大規模な分析用ストレージでは、別の要件が加わります。正しさだけでは足りません。エンジンは、不要な作業を避けるためのメタデータも必要とします。

そこで物理メタデータが登場します。

本記事は [データ型が契約へ変わるとき](/ja/blog/metadata/part2-schema-contracts/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの焦点は、Parquet という名前そのものや、列指向フォーマットの一般論ではありません。実際の問いは、ファイルに埋め込まれたメタデータが、多くのデータバイトをデコードする前に、スキャンコスト、プルーニング、相互運用性、実行形状にどう影響するかです。

## ストレージが必要とするのは、正しさのためのメタデータだけではありません

スキーマ契約があれば、`created_at` がタイムスタンプであり、`country_code` が2文字の文字列であることは利用側に伝えられます。それは解釈や互換性のために重要です。しかし、日付フィルタに対してファイルの 98 パーセントを飛ばせるかどうかまでは教えてくれません。

そこが、論理メタデータから物理メタデータへの段差です。

論理メタデータは、次のような問いに答えます。

- どんなフィールドが存在するか
- 各フィールドの型は何か
- どのフィールドが必須で、どれが任意か
- 下流システムが依存する意味や互換性保証は何か

物理メタデータは、別の問いに答えます。

- ファイルのどこに各列が保存されているか
- row group はいくつあるか
- どのエンコーディングと圧縮が使われたか
- 列セグメントごとの min/max は何か
- データページを読む前に row group を丸ごと飛ばせるか

CSV や JSON と比べると、この違いははっきりします。これらの形式はレコード構造を表せても、バイトがストレージに置かれたあと、エンジンに構造上の助けをほとんど与えません。CSV リーダーは通常、線形に走査して行をトークン化し、そのあとで値を解釈しなければなりません。JSON はネスト構造を持てますが、どこに列があり、ページ境界がどうなっていて、`event_date >= '2026-05-01'` の条件でどのセグメントを丸ごと飛ばせるかを教えるコンパクトなフッターは持ちません。

[Parquet](https://parquet.apache.org/) は、その欠けていた層のために設計されました。データを列指向レイアウトで保存しつつ、エンコード済みの列データに対して、クエリエンジンが素早く参照できるメタデータを組み合わせます。ファイルには値だけが入っているのではありません。その値をどう見つけ、どう飛ばし、どうデコードできるかという計画も一緒に入っています。

## ファイルレイアウト

Parquet のファイル構造は、ファイル末尾から逆向きに捉えると理解しやすくなります。

重要なメタデータはフッターにあります。リーダーは通常、オブジェクト末尾付近を小さく range read し、フッター長を見つけ、そのあとシリアライズされたメタデータブロックを読みます。そのブロックには、スキーマ、row group の一覧、列チャンクごとのメタデータ、エンコーディング、圧縮情報、オフセット、統計情報が入っています。

頭の中での簡略図は、次のようになります。

```text
PAR1
  column data pages
  column data pages
  ...
  row group N column chunks
  serialized footer metadata
footer_length
PAR1
```

実際の wire format はもっと細かいですが、運用上は、フッターがエンジンに対してファイルへの索引を提供していると考えれば十分です。

主な構造単位は次のとおりです。

- Row group: ひとつのファイル内にある、行の水平方向のまとまり
- Column chunk: ひとつの row group の中で、ある列に対応するバイト列
- Page: column chunk 内のより小さな単位で、データや辞書の格納に使われる
- Footer metadata: それら全体がどう組み合わさっているかを記述するファイルレベルの目録

実際には、フッターには少なくとも次の4種類の情報が含まれていることが多いです。

- ファイルのスキーマメタデータ
- row group ごとの行数とバイトサイズ
- 列チャンクごとのオフセット、エンコーディング、圧縮方式、統計情報
- エンジンが無差別にフルスキャンせず、range read を計画できるだけの構造情報

最後の点は見落とされがちです。オブジェクトストレージでは、クエリがファイル全体の 3 パーセントだけ必要なのか、90 パーセント必要なのかを知るためだけに、ファイル全体をダウンロードしたくありません。フッターがあることで、安価な計画フェーズが可能になります。エンジンはファイル末尾近くを小さく読み、メタデータをデコードし、そのあと必要な列チャンクに対応するバイト範囲だけを要求できます。この設計は現代の分析用ストレージと相性がよく、ファイルメタデータの調査は、完全なデコードよりずっと少ないネットワークコストと CPU コストで済みます。

たとえば、`order_id`、`customer_id`、`created_at`、`country_code`、`amount_jpy` の5列を 1,000 万行持つファイルを考えます。ライターはこれを 200 万行ずつ、5つの row group に分割するかもしれません。各 row group の内部では、各列が独立した column chunk として書き出されます。フッターには、各 chunk の開始位置、サイズ、圧縮方式、収集された統計情報が記録されます。

この構造があるから、Parquet リーダーは行指向のリーダーのようには動きません。`customer_id` をまったく使わないクエリなら、`created_at` と `amount_jpy` の chunk だけを読む判断ができます。さらに先に統計情報を見て、ある row group は条件に一致し得ないと判定することもできます。

Page によって粒度はさらに細かくなります。column chunk はひとつの巨大な塊ではありません。オプションの dictionary page と、それに続く data page 群から成ることが多いです。この区切りは、ライターにもリーダーにも有用なトレードオフの境界になります。フッターはエンジンを正しい chunk まで導き、page 構造はその chunk の内部で値をどうデコードするかを決めます。Parquet のメタデータが複数の層で効いていることが分かります。ファイル計画、row-group のプルーニング、列投影、ページレベルのデコードです。

## プルーニング

Parquet のメタデータの中で、もっとも即効性の高い価値を持つのは統計情報です。

row group や page の単位で、ライターは次のような値を記録できます。

- 最小値
- 最大値
- null 件数
- 実装や周辺システムによっては distinct 件数

`event_date` に対して 3 つの row group があるファイルを考えます。

```text
row group 1: min=2026-05-01, max=2026-05-07
row group 2: min=2026-05-08, max=2026-05-14
row group 3: min=2026-05-15, max=2026-05-21
```

ここで次のフィルタを実行します。

```sql
SELECT count(*)
FROM events
WHERE event_date >= DATE '2026-05-15';
```

エンジンは、多くの列データを読む前にフッターメタデータを参照できます。row group 1 と 2 は条件を満たし得ないので除外できます。読む必要があるのは row group 3 だけです。

同じことは数値範囲にも当てはまります。

```text
column: amount_jpy
row group 1: min=100, max=1200
row group 2: min=1300, max=4500
row group 3: min=5000, max=25000
```

`WHERE amount_jpy > 10000` なら、最初の 2 グループはすぐに飛ばせます。

これはよく predicate pushdown と説明されますが、実際に有用なのは、どこでその判断が行われるかです。エンジンは、リモートデータベースサーバーへ predicate を押し込んでいるわけではありません。ファイルメタデータを使って、どの物理セグメントを読む価値があるかを決めています。

null 件数も効きます。クエリが `WHERE coupon_code IS NOT NULL` を要求しており、ある row group の null 件数がその row group の全行数と同じなら、そのセグメントは何も返さないので飛ばせます。

パーティションディレクトリは、Parquet のメタデータの代わりではなく、並んで使われることが多いです。データセットがたとえば次のようなオブジェクトストレージパスで整理されているかもしれません。

```text
s3://warehouse/events/dt=2026-05-21/part-0001.parquet
```

ディレクトリベースのパーティショニングは、パスフィルタによってファイル丸ごとの除外を可能にします。そのうえでフッター統計が、残ったファイルの内部をさらに絞ります。前者はデータセットに隣接するメタデータで、後者はファイル埋め込みのメタデータです。現実のエンジンは、両方を併用するのが普通です。

プルーニングの質は、ファイルがどう書かれたかに依存します。行が無秩序に追記されていると、min/max の範囲が大きく重なり、すべての row group が一見関連ありに見えてしまいます。日付、テナント、リージョンのような、よくフィルタされる列でデータがある程度クラスタリングされていれば、統計情報はずっと選択的になります。Parquet が自動で価値を生み出すわけではありません。ライターやテーブルメンテナンスのワークフローが、メタデータの情報量を保てるだけの局所性を維持する必要があります。

## 型

Parquet の型システムも、メタデータが装飾ではなくなる場所のひとつです。

Parquet は、物理型と論理型を区別します。物理型は、値が実際にどう保存されるかを表し、論理型は、読み手が復元すべき高レベルの意味を表します。

たとえば次のような対応があります。

- `INT32` は、Unix epoch からの日数として日付を物理的に保存できる
- `BYTE_ARRAY` は、UTF-8 文字列を物理的に保存できる
- `INT64` は、特定単位のタイムスタンプを物理的に保存できる
- `FIXED_LEN_BYTE_ARRAY` は、10進の小数値を物理的に保存できる

具体例として、小数金額を見てみます。

```text
logical type: DECIMAL(12,2)
physical type: FIXED_LEN_BYTE_ARRAY(8)
```

日付なら、次のような対応になります。

```text
logical type: DATE
physical type: INT32
```

なぜこの区別が重要なのでしょうか。ストレージ効率と相互運用性が、別方向の要求だからです。

ストレージ側から見ると、少数の物理プリミティブに絞ることで、エンコーディングやベクトル化処理を現実的に保てます。相互運用性の側から見ると、読み手は意図された意味を復元できるだけのメタデータを必要とします。システムが `INT32` しか見えなければ、それが件数なのか日付なのか enum 的なコードなのか分かりません。論理注釈がその溝を埋めます。

これが、Parquet のメタデータが単一エンジンを超えて重要になる理由のひとつです。ファイルは Spark で書かれ、後から DuckDB でスキャンされ、Arrow ライブラリ経由で読まれ、Trino で問い合わせられるかもしれません。そうしたシステムはランタイム内部のメモリ構造を共有していませんが、物理メタデータと論理メタデータが十分明確に保たれていれば、同じ解釈に収束できます。

メタデータが曖昧だと、可搬性は下がります。タイムスタンプの意味論は典型例です。物理的な保存形式だけでは、その値を UTC 正規化された時刻として扱うべきか、ローカル時刻として扱うべきか、どの精度境界で解釈すべきかまでは完全に決まりません。少しずつ前提の異なるエンジンは、同じファイルを読んでいても微妙に違う意味へ辿り着くことがあります。

## エンコーディング

Parquet のメタデータは、値がどうエンコードされ、どう圧縮されたかもリーダーへ伝えます。

ライターは、データの特性に応じて dictionary encoding、run-length encoding、plain encoding などを選べます。そのうえで Snappy、Zstandard、Gzip などの圧縮方式を列ページに適用できます。

これらの選択は、実行時の挙動へ直接影響します。

dictionary encoding は分かりやすい例です。`country_code` が少数の繰り返し値しか持たないなら、コンパクトな辞書と整数参照の組み合わせで、ストレージサイズを削減し、スキャン時の局所性も改善できます。フッターや page のメタデータは、どのエンコーディングが使われているかをリーダーへ伝え、正しいデコードを可能にします。場合によっては、行を完全に復元する前に、辞書内容に対して効率よくフィルタを適用することもできます。

圧縮メタデータが重要なのも同じ理由です。3列だけ投影するか、30列投影するかを決めることは、どれだけの圧縮済み chunk を取得し、解凍するかを決めることでもあります。Parquet は列指向なので、圧縮コストはレコード全体ではなく、選択された列と row group に結び付きます。

この点でも、CSV や JSON との比較は有効です。行指向のテキスト形式では、投影の恩恵は限定的です。多くの場合、ファイルの大部分に触れてパースしなければなりません。Parquet では列メタデータがあるため、選択的アクセスは努力目標ではなく、通常の実行パスになります。

## Arrow

Parquet は、[Arrow](https://arrow.apache.org/) と合わせて見ると重要性がよく分かります。

Parquet は、永続化されたコンパクトな列指向ファイルのためのストレージフォーマットです。Arrow は、効率的な分析処理とランタイム間の受け渡しのためのインメモリ列指向フォーマットです。解いている問題は違いますが、どちらも列アクセスと型付き配列を軸にしているため、よく噛み合います。

よくある読み取りパスでは、エンジンがまず Parquet のメタデータを見て必要な row group と列を選び、該当するページをデコードし、それを Arrow 的な列ベクトル、あるいはそれに相当する内部バッファに実体化します。この受け渡しは偶然ではありません。Parquet のメタデータによって、エンジンは不要なデータをデコードせずに済み、Arrow 互換のベクトルレイアウトによって、残ったデータをバッチ単位で処理できます。

ここで、物理メタデータは実行インフラになります。

クエリが `created_at`、`country_code`、`amount_jpy` だけを見るなら、エンジンはファイルレベルで無関係な列を飛ばせます。統計が「条件に一致するのは 2 つの row group だけだ」と示せば、残りを飛ばせます。残ったページは連続したベクトルにデコードされ、filter、aggregate、hash probe のようなベクトル化演算に自然に流し込めます。

結果として減るのは I/O だけではありません。実行形状そのものが変わります。何がベクトル化パスへ入るのかを、メタデータが決めているからです。

具体的な分析クエリを見てみます。

```sql
SELECT country_code, sum(amount_jpy)
FROM orders
WHERE created_at >= TIMESTAMP '2026-05-01 00:00:00'
  AND created_at < TIMESTAMP '2026-06-01 00:00:00'
GROUP BY country_code;
```

ある程度整った Parquet データセットなら、エンジンはよく次のような段階的な判断をします。

1. パーティションメタデータがあれば使い、対象月の外にあるファイルを除外する。
2. 残ったファイルの Parquet フッターを読む。
3. `created_at` の row-group 統計を使い、範囲外のグループを飛ばす。
4. テーブル全体ではなく、`created_at`、`country_code`、`amount_jpy` だけを投影する。
5. 選択されたページを列ベクトルへデコードし、その上で filter と aggregate を実行する。

重要な仕事の大半は、値を完全に実体化する前に行われます。ここがシステムとしての要点です。メタデータは、文書化や相互運用の助けをしているだけではありません。そもそもどのバイトを計算対象にするかを決めています。

これは、Parquet のメタデータが OLTP システムの二次索引と同じではないことも示しています。エンジンは可変なインデックス木を使ってポイントルックアップしているのではなく、選択されたファイルセグメントを走査しています。ただし分析ワークロードでは、その「投影・プルーニング・バッチデコード」の組み合わせこそが、書き込みの単純さと読み取り効率のちょうどよい妥協点になることが多いのです。

## トレードオフ

Parquet のメタデータは強力ですが、魔法ではありません。

トレードオフは大きく2種類あります。ひとつは意味上の境界に関するものです。ファイルフォーマットが何を信頼できる形で表現できて、何を上位のメタデータシステムへ逃がす必要があるか。もうひとつは実行品質に関するものです。メタデータが、プルーニング、計画、可搬な性能をどれだけ支えられるかです。

### 意味上の境界

このあたりで、Parquet の限界もはっきり見えてきます。

チームは、列に型情報以上のものを持たせたくなります。たとえば「値引き後の総注文金額（円）」や「東京ローカル時刻で記録された顧客登録時刻」のような、人間向けの説明です。これらは、単なる文書品質ではなく解釈に影響するので、本物のメタデータ要件です。

Parquet はある程度のスキーマ意味を持てますが、汎用的なフィールド説明フォーマットではありません。移植性のあるコア仕様では、列は名前、物理型、繰り返し規則、任意の論理型注釈を持てます。しかし、多くのエンジンで共通に理解される、データベースの column comment のような標準化された first-class の説明フィールドは持っていません。

この違いは運用上重要です。エンジンやライブラリによっては、書き込み時に独自メタデータを付けられますし、ファイルレベルの key-value メタデータもあります。埋め込み Arrow スキーマや、カタログ、レイクハウス形式の外部テーブルメタデータのように、隣接するスキーマ層を通じてより豊かな注釈を保つこともあります。ただし、それは「生の Parquet 自体が、ポータブルな列説明の信頼できる保存先である」と言うのとは違います。

ですから、「Parquet は列の説明を持てるか」と問われたときの実務的な答えはこうです。すべてのエンジンが公式な列コメントとして解釈する標準的な方法は、基本的にありません。説明文がツール、レビュー工程、データプロダクトをまたいで確実に残る必要があるなら、テーブルスキーマ、カタログ、レジストリ、あるいは意味文書化に向いた別のメタデータ層に置く方が安全です。

タイムゾーンの扱いにも同じ境界があります。

Parquet は、素の文字列や整数よりはタイムスタンプの意図を表現できますが、すべてのタイムゾーン問題を解決するわけではありません。現代の Parquet の論理型では、タイムスタンプの単位や、値が UTC に調整済みかどうかを表せます。これにより、ある値が「瞬間」を表すタイムスタンプなのか、ローカルタイムスタンプ的なものなのかを読み手が区別しやすくなります。

ただし、`Asia/Tokyo` のような名前付きタイムゾーンを、標準の論理型パラメータとして列に保存できるわけではありません。言い換えると、Parquet は「このタイムスタンプは UTC 正規化済みか」「UTC 調整されていないか」を表す助けにはなりますが、あるソースシステムが JST や America/Los_Angeles を解釈の基準にしているという業務ルールまで、普遍的にエンコードする場所ではありません。

そのため、チームの実務パターンは次のようになります。

- Parquet の論理タイムスタンプ注釈で、単位と UTC 調整の意味を表す
- 名前付きタイムゾーンのルールは、隣接メタデータやデータ契約に保存する
- 行ごとに元タイムゾーンが分析上重要なら、ファイルメタデータに期待せず、データとして明示的に持つ

これはシリーズ全体の流れとも一致します。物理メタデータは、データを見つけ、飛ばし、デコードすることには非常に優れています。一方で、業務上の意味を完全に運ぶ媒体としては弱い。豊かな説明、所有者、単位、明示的なタイムゾーン規約まで必要になると、Parquet 単独ではなく、より上位のメタデータシステムと組み合わせる必要があります。

### 実行品質

最初の運用上のトレードオフは、統計情報の質です。プルーニングが効くのは、収集されたメタデータが存在し、しかも役に立つときだけです。統計が欠落していたり、切り捨てられていたり、値の混ざり方が広すぎる row group が作られていたりすると、フィルタの選択性は下がります。Parquet としては妥当なファイルでも、プルーニング対象としては poor になり得ます。

2つ目は、ファイルサイズと row-group サイズです。極端に小さいファイルは計画オーバーヘッドやオブジェクトストレージのリクエストコストを増やします。極端に大きい row group は選択性を弱め、狙った predicate に対して重い読み取りを強います。完璧な単一サイズはありません。最適値は、ワークロードの形、オブジェクトストレージの特性、下流エンジンの期待によって変わります。

3つ目は、可搬性とエンジン固有最適化のバランスです。Parquet は相互運用性を意図して設計されていますが、すべての最適化が同じように移植可能なわけではありません。特定の統計、page index、ライターの癖に強く依存するエンジンもあります。あるエンジン向けに最適化されたデータセットは、どこでも読めるまま、別のエンジンでは性能が不均一になることがあります。

物理的には妥当でも、意味上のリスクが潜むこともあります。ある列がよくエンコードされ、豊かな注釈を持っていても、下流の読み手を混乱させる形で意味が変わることはあり得ます。論理メタデータは助けになりますが、ガバナンスや移行の規律を不要にするわけではありません。物理効率は意味の正しさの代わりにはなりません。

もうひとつ微妙な失敗モードがあります。エンジニアが「Parquet だから速い」と前提し、実際にファイルがどう書かれているかを見なくなることです。しかしスキャン挙動は、拡張子ではなく、生成されたファイルの具体的なメタデータに依存します。ソートが悪い、row-group の範囲が広すぎる、論理注釈が不統一、不要な列重複がある。こうしたことだけで、期待していた利点は簡単に損なわれます。

## 単一ファイルの外へ

ここまでで、メタデータはフィールドを説明する以上のことをしています。

フッターはリーダーにファイル構造を伝え、row group と列チャンクの統計はどのセグメントを飛ばせるかを決め、型注釈はエンジン間で意味を保ち、エンコーディングと圧縮メタデータはデコードコストを左右し、Arrow に沿った読み取りパスはそれらの判断をベクトル化実行へつなげます。

これは第2回からの大きな変化です。第2回では、メタデータは正しさのために提供者と利用者を調停しました。ここでは、クエリに必要な計算量まで決めています。

ただし、Parquet のメタデータのスコープは、あくまでファイル単位と、せいぜいパーティションディレクトリのような近接レイアウトにとどまります。データセットが数千ファイルにまたがると、次の問題が現れます。エンジンは、ファイル集合、スナップショット、パーティション進化、並行更新をまたいで考えられるメタデータを必要とします。

それが第4回への引き継ぎです。

メタデータが多数のファイル、マニフェスト、スナップショットにまたがるようになると、それは単なるファイル索引ではありません。データセットそのもののための分散協調レイヤーになります。
