第3回は、メタデータが単一ファイルをはみ出した地点で終わりました。

Parquet のフッターは、ひとつのオブジェクトがどうレイアウトされ、どの row group 統計があり、どの列チャンクを読む価値があるかを教えてくれます。それ自体は強力ですが、その効力はそのファイルの境界の中だけです。現代の分析データセットがひとつのファイルであることはほとんどありません。通常は、オブジェクトストレージ上の数千の Parquet オブジェクトで構成され、独立したジョブが時間差で書き込み、複数のエンジンが問い合わせ、現実の並行性のもとで更新されます。

そこで、メタデータはデータセット全体の協調レイヤーになります。

本記事は [Parquetと物理メタデータの台頭](/ja/blog/metadata/part3-parquet-physical-metadata/) と、シリーズ全体の導入である [メタデータ — 列コメントから分散コントロールプレーンへ](/ja/blog/metadata/) を引き継ぎます。ここでの実際の問いは、ひとつのファイルをどう効率よくスキャンするかではありません。多数のファイルに分散したテーブルを、安全に変更しながら、複数エンジンに対して安定して効率よいデータビューをどう提供するかです。

## ファイルメタデータはテーブルメタデータではありません

第3回は、ファイルメタデータに焦点を当てました。

ファイルメタデータが答えるのは、次のような問いです。

- このファイルにはどんなスキーマが埋め込まれているか
- どんな row group と列チャンクが存在するか
- どんな min/max 統計があるか
- 選択した列をデコードするにはどのバイト範囲を取得すべきか

これらはローカルな問いです。ひとつのオブジェクトの内部でのスキャン計画に役立ちます。

レイクハウス系システムには、別の層が必要です。そこで答えるのは、次のようなテーブルの問いです。

- 現在どのファイルがこのテーブルに属しているか
- クエリはどのスナップショットのテーブルを読むべきか
- どのスキーマとパーティション仕様が現行か
- 2つのコミットのあいだで何が変わったか
- 2つのライターは、競合するテーブル状態の公開をどう防ぐか

この違いは重要です。オブジェクトストレージはネイティブにテーブル意味論を提供しません。オブジェクトストアは blob を堅牢に保持し、range read にも向いていますが、`orders` の現在の一貫したバージョンがどの 14,000 個の Parquet ファイルで構成されるかを教えてはくれませんし、伝統的なデータベースカタログのような原子的なテーブル更新プリミティブも提供しません。

より上位のメタデータ層がなければ、エンジンはしばしば高コストで壊れやすいパターンに頼ります。

- ディレクトリを再帰的に一覧してファイルを見つける
- パス名だけからパーティションを推測する
- 見えているファイルはすべて最新テーブル状態の一部だと仮定する
- 進行中の書き込みや obsolete な書き込みを、アドホックな命名規約で見分ける

こうしたやり方は、スケールすると破綻します。巨大なプレフィックスの listing は高コストです。並行ライターは競合します。部分的な書き換えは、混ざったテーブル状態を露出させます。権威あるバージョン付きテーブル定義がないため、スキーマ変更も追いにくくなります。

[Apache Iceberg](https://iceberg.apache.org/) や [Delta Lake](https://delta.io/) のようなオープンテーブルフォーマットは、この協調問題を解くためにあります。ディレクトリ中のファイル集合をそのままテーブルとみなすのではなく、データファイルを指すバージョン付きメタデータ構造としてテーブルを扱います。

## オブジェクトストレージが協調問題を変えた

Parquet が普及した理由と、レイクハウスメタデータが必要になった理由は分けて考えると分かりやすくなります。

Parquet は分析スキャンのための効率的なファイルレイアウトを解決しました。オブジェクトストレージは、安価で堅牢でスケーラブルな保存先を提供しました。どちらも、それだけではテーブル協調を解決していません。

日次の append パイプラインが次のオブジェクトを書き込むとします。

```text
s3://warehouse/orders/dt=2026-05-19/part-0001.parquet
s3://warehouse/orders/dt=2026-05-19/part-0002.parquet
s3://warehouse/orders/dt=2026-05-20/part-0001.parquet
...
```

テーブルが append-only で、しかも単一エンジンが緩く読むだけなら、ディレクトリ探索でもしばらくは成立するように見えます。

しかし、圧力は4方向からやってきます。

第一に、ファイル数が増えます。クエリは、大きなプレフィックスの下にあるすべてのオブジェクトを一覧し、検査してから live なファイルを見つけるべきではありません。

第二に、更新と削除が出てきます。分析テーブルは、もはや append ログだけではありません。チームはパーティションを書き換え、ファイルを compaction し、GDPR 削除を適用し、CDC ストリームをマージし、過去期間を backfill します。

第三に、複数エンジンが同じテーブルを読みます。Spark、Trino、Flink、DuckDB、独自サービスが、ランタイムメモリを共有しないまま、現在のテーブル状態について共通の理解を必要とします。

第四に、複数ライターが現れます。バッチの compaction ジョブ、ストリーミング取り込みジョブ、backfill プロセスが、ほぼ同じタイミングで変更を公開しようとするかもしれません。

この段階で、本当の問題はファイルエンコーディングではありません。テーブル状態に対する分散協調です。

レイクハウスの答えは、メタデータを権威あるものにすることです。クエリはまずテーブルバージョンを解決し、そのバージョンから関連するファイル集合を導きます。計算はメタデータに従い、その逆ではありません。

## Iceberg

[Apache Iceberg](https://iceberg.apache.org/) は、テーブルをメタデータファイル、スナップショット、manifest list、manifest から成る階層として表現します。

重要なのは、シリアライズ形式の細部ではありません。責務がどう分かれているかです。

- table metadata file は、現在のスナップショット、スキーマバージョン、パーティション仕様、ソート順、テーブルプロパティといったテーブル全体の状態を記述する
- snapshot は、テーブルのひとつのコミット済みバージョンを表す
- manifest list は、その snapshot に属する manifest を記録する
- 各 manifest は、データファイルの集合と、それらのファイルに関するパーティション値、統計、sequence number、状態などのメタデータを記録する

この構造により、Iceberg は「いまこのテーブルに属しているファイルはどれか」という単純だが重要な問いに、スケーラブルに答えられます。

答えは「このディレクトリ配下に見えているファイル全部」ではありません。「現在の snapshot から、manifest list と manifest をたどって到達できるファイル」です。

### フロー

簡略化した Iceberg の読み取りフローを文章で表すと、次のようになります。

1. エンジンがカタログに現在のテーブルメタデータの場所を問い合わせる。
2. 現在の table metadata file を読み、アクティブな snapshot ID を知る。
3. その snapshot の manifest list をロードする。
4. クエリ predicate を満たす可能性がある manifest だけを読む。
5. そこから、実際にスキャンが必要なデータファイル群を導く。
6. そのあとで初めて、選択されたファイルに対する Parquet や ORC の読み取りを計画する。

これは、ディレクトリ listing より大きく改善されています。エンジンは、テーブル所属を再構築するためにオブジェクトストレージ全体を再帰的にたどる必要がありません。テーブルバージョンをすでに記述しているコンパクトなメタデータ経路を読むだけで済みます。

manifest は、クエリ時の協調コストも下げます。大規模データセットの全ファイルフッターを開いて関連性を判断する代わりに、エンジンはまず manifest レベルのパーティション情報やファイル統計を使えます。クエリが `event_date = '2026-05-20'` でフィルタするなら、多数の manifest とデータファイルを、Parquet フッターを読む前に除外できることがあります。

ここで重要な区別は次のとおりです。

- Parquet のメタデータは、ファイル内のプルーニングに効く
- Iceberg のメタデータは、多数のファイルに触れる前にテーブル全体をまたいだプルーニングに効く

### スナップショット

snapshot は、一貫したテーブル状態の単位です。

ライターが新しいデータを append したり、あるファイル集合を書き換えたり、削除を適用したりするとき、見えているテーブル状態をファイル単位で直接 mutation するのではありません。次の snapshot を表す新しいメタデータを準備し、それをカタログ機構を通じて原子的に commit します。

これにより、読み手は安定したビューを得られます。snapshot `N` で始まったクエリは、その直後に `N+1` が commit されても、引き続き `N` を読み続けます。これは、レイクハウステーブルが、データベース内部のページラッチやストレージエンジン実装に頼らず、再現可能な分析読み取りを提供できる理由のひとつです。

snapshot は、テーブル履歴も明示化します。エンジンや運用者は、次のような問いを投げられます。

- 直近の commit で追加されたファイルは何か
- どの snapshot がスキーマ変更を導入したか
- テーブルを過去の一貫した状態へロールバックできるか

これらはファイルフォーマットの問いではなく、メタデータの問いです。

## Delta Lake

[Delta Lake](https://delta.io/) は、トランザクションログを中心とした別のメタデータモデルで、同様の目標に到達します。

Delta は、manifest tree を主軸にテーブル状態を表すのではなく、`_delta_log` 配下の順序付きログとして変更を記録します。JSON の commit file と、定期的な checkpoint が、ファイル追加、ファイル削除、テーブルメタデータ更新、プロトコルバージョン変更といったアクションを保持します。

概念的には、現在のテーブル状態は、あるバージョンまでログを再生し、毎回全履歴を読み直さなくて済むよう checkpoint を参照して導かれます。

中心となる考え方は単純です。テーブルは、生のディレクトリ内容ではなく、順序付きのメタデータアクションの履歴で定義されるということです。

### コミット例

単純化した Delta の append は、次のように考えられます。

ライターはまず新しい Parquet データファイルを書き出します。しかし、アップロードしただけでは、それらはまだテーブル状態の一部ではありません。

そのあとで、次のようなアクションを含む新しいログバージョンの commit を試みます。

```json
{
  "add": {
    "path": "part-00017-...snappy.parquet",
    "partitionValues": { "dt": "2026-05-20" },
    "size": 13428192,
    "stats": "{\"numRecords\":500000,\"minValues\":{...},\"maxValues\":{...}}"
  }
}
```

この commit が成功すれば、そのファイルは次のテーブルバージョンの一部になります。競合する並行更新のために失敗したなら、ライターは新しい最新バージョンを前提に再評価しなければなりません。

だからこそログが重要です。可視性は、Parquet ファイルがストレージに存在するかではなく、メタデータの commit によって制御されます。

### 並行性

Delta Lake は、楽観的並行性制御を採用するとよく説明されます。運用上の意味に引き直すとこうです。ライターは、競合は比較的少ないと仮定し、書き込みパス全体に長い排他ロックを持たずに進み、commit 時に「その前提を無効にする変更がテーブルに入っていないか」を検証します。

たとえば 2 つのジョブを考えます。

- Job A は、昨日の小さなファイルを大きなファイルへ compaction する。
- Job B は、同じテーブルへ遅延到着したレコードを append する。

どちらもテーブルバージョン 120 から開始するかもしれません。もし Job B が先に変更を commit し、その内容が同じ論理スコープに重なるなら、Job A が version 121 を commit しようとした時点で、retry か fail が必要になります。どちらになるかは conflict semantics に依存します。

これは分散システム一般でも見慣れたパターンです。重要なのは、並行性チェックがメタデータにあることです。ストレージ層は immutable なデータファイルを保持し、ログが「どのファイル集合がテーブルを構成するか」を調停しています。

checkpoint があるのは、ログの成長そのものがスケーリング問題になるからです。成熟したテーブルには何千もの commit があるかもしれません。定期的な checkpoint はテーブル状態を materialize し、新しい読み手が最近の圧縮表現と、それ以降のより短いログ差分だけで bootstrap できるようにします。

## Hive Metastore

[Apache Hive](https://hive.apache.org/) と Hive Metastore は、いまでも重要な比較対象です。古いメタデータパターンが何をうまく解いていて、どこから厳しくなるかをよく示してくれるからです。

Hive Metastore は、テーブル、スキーマ、パーティション、ストレージロケーションの中央カタログを提供しました。これは、各エンジンが unmanaged なパス規約を独自に発明していた時代と比べれば大きな前進でした。

問題は、Hive Metastore が無価値だったことではありません。問題は、それが別の圧力分布に対して設計されていたことです。

古典的な external table の使い方では、パーティションディレクトリが metastore に登録され、実際のファイル一覧は暗黙にストレージ側に置かれることが多いです。現代のワークロード圧力の下では、この形にはいくつかのギャップがあります。

- テーブルカタログはパーティションパスを知っていても、ある snapshot に対する権威ある全ファイル集合は持っていないことがある
- 更新や削除は、テーブル全体で自然に transactional とは言い難いパス書き換えに依存しがち
- パーティション数が非常に大きくなり、管理コストが上がる
- エンジンは各パーティションの下で依然として重い listing を必要とすることがある
- スキーマ進化やパーティション進化は、新しいオープンテーブル形式より扱いにくい

このため、Hive Metastore はここでは対照例として捉えるのが適切です。ある程度のメタデータは中央集約しますが、Iceberg や Delta が必要とする snapshot 指向の協調モデルまでは、それ単独では提供しません。

多くの本番環境では、今でも metastore 的なサービスや、より新しい catalog API が入口として使われます。違いは、そのカタログが何を指しているかです。レイクハウス設計では、カタログは complete な協調ロジックそのものではなく、バージョン付きテーブルメタデータ構造を解決する入口になります。

## 進化

メタデータがテーブルのコントロールプレーンになると、進化ルールは明示的な運用課題になります。

もっとも分かりやすいのはスキーマ進化です。列追加は、パーサ変更だけではありません。テーブルメタデータが新しいスキーマバージョンを記録し、フィールドの同一性や進化意味論を正しく保ち、古いファイルと新しいファイルを読み手が整合的に扱えるようにする必要があります。

この意味で、Iceberg の field ID は運用上とても重要です。rename を、同じ論理列へのラベル変更ではなく、「古い列の削除と新しい別列の追加」と誤解してはいけません。安定した field ID があることで、名前一致だけに頼るより安全に、エンジンやメンテナンスジョブが進化を扱えます。

パーティション進化も同じくらい重要です。実際のテーブルは、最初に選んだパーティション設計をやがて超えます。日単位パーティションで始まったデータセットが、月単位、bucket transform、タイムスタンプ列に対する hidden partition transform を必要とするかもしれません。

古いレイアウトでは、パーティションパス自体が定義になりがちです。すると変更は痛みを伴います。ストレージディレクトリの形そのものが契約の一部になるからです。現代のテーブル形式は、論理的なパーティション仕様を生のディレクトリ名からある程度切り離し、テーブルを進化させても問い合わせ可能に保ちます。

原子的なテーブル状態変更は、その裏側のもう半分です。

あるメンテナンスジョブが 800 個の小ファイルを 20 個の大きなファイルへ書き換える場合、読み手が「古いファイルの一部は消えたが、新しいファイルは一部しか見えていない」という半端な状態を観測してはいけません。変化全体が、ひとつの snapshot あるいはログバージョンから次への単一のメタデータ遷移として現れる必要があります。

これが、レイクハウスメタデータにおける分散システム的な核心です。immutable なデータファイルと、原子的なテーブル状態の公開です。

## 運用

整ったメタデータモデルは必要ですが、それだけで運用圧力が消えるわけではありません。むしろ、圧力を見える形にしてくれます。

もっとも典型的なのは small files です。数十万の小さなオブジェクトを持つテーブルは、リクエストオーバーヘッドを増やし、manifest やログを膨らませ、スキャン効率を下げます。そのため compaction は、ストレージの掃除だけではありません。メタデータ保守でもあります。ファイル集合が、読み手にもテーブルメタデータ層にも扱いやすくなります。

commit contention もよくある問題です。多数の独立ジョブが同じ hot table を狙うと、楽観的並行性制御は頻繁な retry に劣化し得ます。正しさは保たれても、スループットとレイテンシは落ちます。これはフォーマットの問題であると同時に、ワークロード設計の問題でもあります。重なったテーブル状態を、あまりに頻繁に、あまりに多くのライターが公開しようとしているのです。

metadata bloat は静かですが重要なリスクです。snapshot は積み上がり、manifest は増え、トランザクションログは伸びます。checkpoint、snapshot expiration、manifest compaction は助けになりますが、データパスと同じように、テーブルのメタデータパスも定期保守が必要です。

catalog 依存も重要です。テーブル形式が open であっても、クエリは依然として catalog 解決パス、正しい権限、整合した metadata location に依存します。catalog が落ちている、設定不備がある、状態が stale である。その場合、ストレージに安全に置かれたデータファイルは、自動的には利用可能なテーブルになりません。

だからこそ、メタデータを単なる説明のオーバーヘッドとして語るのはやめるべきです。レイクハウス規模では、メタデータサービスは serving path の一部です。

## 計画

もっとも重要なシステム上の帰結は、計算が始まる前に何が起きるかです。

クエリエンジンは、まずプレフィックス配下のすべてのファイルをダウンロードし、あとで最適化しようとはしません。健全なレイクハウス設計では、メタデータを段階的にたどって考えます。

エンジンはおおむね次のような流れを取ります。

1. カタログでテーブルを解決する。
2. 現在の snapshot あるいはログから導いたテーブルバージョンを読む。
3. manifest、ログ、パーティション、checkpoint のメタデータを使って無関係なファイルを除外する。
4. 残った候補について、Parquet 統計のようなファイルレベルメタデータを読む。
5. そのあとで初めてスキャンと計算タスクをスケジュールする。

この流れは、snapshot と manifest がなぜ高価な listing や協調作業を減らすかを説明しています。クエリが「同じテーブル」と思っている対象に対して、テーブル所属と変更履歴のコンパクトで権威ある要約を与えてくれるからです。その結果、計画フェーズはストレージクロール主導ではなく、メタデータ主導で進められます。

同時に、ファイルメタデータとテーブルメタデータの境界も明確になります。

ファイルメタデータは、「このファイルの中には、こうした row group と列チャンクがあり、こうした統計がそれらを説明している」と言います。

テーブルメタデータは、「この一貫したテーブルバージョンでは、これらのファイルが live であり、このスキーマが有効であり、このパーティション仕様が現行であり、この変更履歴が現在状態を生んだ」と言います。

前者がなければスキャンは無駄が多くなります。後者がなければテーブルそのものが不安定になります。

だから、オープンテーブルフォーマットはメタデータを分散システムの関心事へ押し上げました。オブジェクトストレージ上でテーブルが多数の immutable file にまたがると、正しさも効率も、複数エンジンが信頼できる権威あるメタデータ遷移に依存するからです。

第5回は、そこから先を扱います。

カタログ、snapshot、ファイル集合が解決されたあとでも、クエリエンジンは「そもそもどんな計算を走らせるべきか」を決めなければなりません。その次の段階でも、メタデータが中心にあります。統計情報、プルーニング規則、計画入力が、最初のフルスキャンより前に実行形を決めます。
