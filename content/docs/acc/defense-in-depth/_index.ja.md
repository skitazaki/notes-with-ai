アクセス制御は、アイデンティティと認可を取り巻く追加制御によって、誤設定・誤用・侵害の影響を抑えられるときに最も強くなります。

## 概要

多層防御は、単一の制御では十分でないことを認める考え方です。認証はフィッシングされ得ますし、ポリシーは誤設定され得ます。トークンは盗まれ、ワークロードは侵害され、運用者は誤承認することもあります。多層防御は、攻撃者に複数の制御領域を越えさせることで影響範囲を下げます。

アクセス制御の文脈では、強い認証、デバイス信頼、ネットワーク分離、条件付きアクセス、実行時認可、行動分析、ログ、インシデント対応が主な層になります。

## 主要概念

### 多層の適用

高リスク要求は、しばしば次の複数ゲートを通ります。

1. 本人確認と MFA
2. デバイス / ワークロード信頼の評価
3. ネットワークとセッションポリシー
4. 要求された操作に対する認可ポリシー
5. 異常検知とトランザクション監視
6. ログ、アラート、封じ込め制御

各層は別々の問いに答えます。これらを組み合わせることで、認可だけに過剰な責務を背負わせずに済みます。

### フェイルオープンとフェイルクローズ

重要システムでは、依存先が使えないときにどう振る舞うかを決める必要があります。fail-closed は安全性を高めますが可用性を損なう可能性があります。fail-open は継続性を守りますが重大な露出を生み得ます。正解は、事業重要度、安全性への影響、補完制御によって変わります。

### 緊急時アクセス

緊急時アクセス（Break-glass access）は現実の運用に必要ですが、例外でなければなりません。緊急用の経路には、明確な範囲、時間制限、強いログ、事後レビュー、可能なら帯域外承認が必要です。

## 実装と運用

### 実務的な制御スタック

- 強い認証とフィッシング耐性 MFA
- デバイス状態評価またはワークロード証明
- 利用者経路、サービス経路、管理経路の分離
- アプリケーション / API 層での文脈依存認可
- サンドボックス化や外向き通信制御などの実行時封じ込め
- SIEM 連携と対応手順書

### 緊急時アクセスの安全な運用

緊急時アクセスは、隔離されたアイデンティティ、事前定義された狭い範囲、自動的に強制される短い有効期限を前提に運用すべきです。申請経路には、誰が起動したか、なぜ通常経路では足りなかったか、どのシステムに触れたか、帯域外承認があったかを残します。

利用後は、即時レビュー、必要に応じた資格情報ローテーション、標準の制御経路がなぜ失敗したのかの原因確認まで含めるべきです。緊急時アクセスが頻繁に使われているなら、それは通常のアクセスモデルか承認待ち時間が壊れている兆候です。

### 監視すべきこと

- 異常な権限昇格やトークン発行
- ポリシー回避の試み
- 人のアイデンティティに対する不自然な移動や地域変化
- 通常と異なるサービス間呼び出し経路
- 拒否された操作が続いた直後の特権操作成功

多層防御が有効になるのは、アラートをアイデンティティ、ポリシー、回復アクションへ結び付けられる場合だけです。
