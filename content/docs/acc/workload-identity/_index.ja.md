マシン ID は、現代のシステムに人間の運用者よりもはるかに多くのサービス、ワークロード、パイプライン、デバイスが存在するため、アクセス制御アーキテクチャの中でも急速に重要性を増している領域です。

## 概要

ワークロードアイデンティティは、「あるマシンが別のマシンを何者として認識し、どの条件で信頼するか」という根本問題に答えます。従来は静的シークレットや長寿命サービスアカウントに依存しがちでしたが、現代では証明、短命資格情報、証明書ベースのアイデンティティ、厳密に絞られたサービス間認可が中心になります。

この領域は、Kubernetes、サーバーレス、CI/CD、API、サービスメッシュ、ロボティックオートメーション、IoT にまたがります。課題はワークロードを認証することだけではありません。その実行役割に必要最小限のアイデンティティと権能を与えることです。

## 主要概念

### 主なアイデンティティ形態

マシン ID は主に次の形で現れます。

- サービスアカウントやクラウドロール
- X.509 証明書と mTLS アイデンティティ
- ワークロードフェデレーション用の OIDC / OAuth トークン
- 署名付きワークロード証明
- ハードウェアに根差したデバイスアイデンティティ

### なぜ静的シークレットは破綻しやすいか

長寿命の共有シークレットは、影響範囲を広げ、属性把握を弱くし、ローテーションを困難にします。複数サービスが同じ資格情報を提示するため、どのワークロードが動いたのかも曖昧になります。

証明、ローテーション、ワークロードメタデータに結び付いたポリシーチェックを伴う自動発行の短命資格情報は、封じ込めと可観測性を改善します。

### SPIFFE / SPIRE とワークロード信頼

SPIFFE / SPIRE のような枠組みは、標準化された識別子と発行フローでワークロードアイデンティティを定式化します。価値は相互運用性だけではありません。アイデンティティを検証可能なワークロード文脈に明示的に束ねることにあります。

## 実装と運用

### 推奨パターン

- クラウドが対応しているならフェデレーテッドなワークロードアイデンティティを使う
- 広すぎるサービスアカウントを狭い実行時アイデンティティに置き換える
- pod、namespace、cluster、image signature、build provenance などの証拠にアイデンティティを結び付ける
- ワークロードの認証と、その操作の認可を分離する
- 資格情報は自動ローテーションし、発行異常を監視する

### 高リスク領域

- 広い本番権限を持つ CI/CD システム
- 複数業務で使い回される共有ロボット / バッチアイデンティティ
- 期限切れしない API トークン
- 過剰権限ワークロードにマウントされた Kubernetes サービスアカウント
- 失効や証明の仕組みを持たないデバイス群

運用上の目標は、マシン間信頼をシークレット配布の問題から、アイデンティティとポリシーの問題へ変えることです。
