良いアクセス制御設計は、普遍的な正解よりも、明示的なトレードオフ管理に依存します。

## 概要

RBAC と ABAC、JWT と opaque token、集中型と分散型の認可、長寿命資格情報と短命資格情報のような選択は、それぞれ異なる運用現実を最適化します。機能比較だけで判断すると誤りやすく、実際には拡張性、失敗モード、ガバナンスコスト、移行負荷、信頼境界の形を含めて比較する必要があります。

## 主要概念

### 典型的な比較軸

- **RBAC vs ABAC**: 安定運用と動的精度
- **集中型認可 vs 分散型認可**: 一貫性とローカル自律性
- **JWT vs opaque token**: ローカル検証とサーバー側の強い失効制御
- **静的認可 vs 継続的認可**: 低い実行コストと変化するリスクへの追従
- **長寿命資格情報 vs 短命資格情報**: 運用容易性と封じ込め・属性把握

### 先に問うべきこと

1. 監査人、運用者、開発者に何を説明しなければならないか
2. 判断時点で鮮度が必要なデータは何か
3. 許容できる失敗は何か: 遅延、非整合、可用性低下のどれか
4. ポリシー意味論と展開の安全性は誰が持つか
5. 最初のモデルが足りなかった場合の移行はどれほど困難か

## 実装と運用

### 判断のヒューリスティクス

- 信頼境界に合う範囲で最も単純なモデルを優先する
- 実行時に本当にリスクが変わる場合にだけ動的性を追加する
- 意味論のずれを許容できないならポリシーロジックを全チームに分散しない
- 既存 RBAC の周囲に文脈条件を足すなど、可逆的な移行経路を優先する
- 理論上の表現力ではなく運用コストを測る

### JWT と opaque token の運用モデル

JWT は、多数のサービスが低遅延でローカル検証したい場合に向いています。その代わり、失効、クレームの最小化、署名鍵ローテーション、audience の厳密化が重要になります。中央への問い合わせなしに多くのコンポーネントが受け入れ得るからです。

opaque token は、イントロスペクションや参照なしでは意味を持たないため、制御を認可サーバーやゲートウェイ側へ戻しやすくなります。サーバー側での失効やクレーム変更には有利ですが、オンライン検証、キャッシュ方針、障害時の扱いに依存します。そのため大規模環境では、エッジでは JWT を使ってスケールを確保し、高リスクな内部委譲経路では opaque token や reference token を使う構成もよく見られます。

トレードオフ文書が有用なのは、設計意図を保存できるからです。なぜその選択をしたのかが残っていれば、場当たり的な例外でアーキテクチャが崩れにくくなります。
