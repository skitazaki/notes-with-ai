アーキテクチャパターンは、繰り返し現れる適用と信頼分配の問題に対する再利用可能な答えです。

## 概要

パターンが重要なのは、アクセス制御が個別アルゴリズムの品質よりも、統合点で失敗しやすいからです。再利用可能なパターンは、ポリシーがどこにあり、判断がどう伝播し、どこで信頼前提が危険になるかを整理する助けになります。

## 主要概念

### 集中型認可

中央のポリシーサービスが多数のアプリケーションの判断を行います。一貫性とガバナンスは強まりますが、遅延、可用性、ポリシー配布が論点になります。

### 組み込み型認可

アプリケーションがプロセス内でポリシーを評価します。自律性とローカル耐障害性は高まりますが、意味論が標準化されていないとポリシードリフトを招きます。

### サイドカー / サービスメッシュ適用

プロキシやサイドカーがワークロードの近くでアイデンティティと粗粒度ポリシーを強制します。サービス間制御、mTLS、統一テレメトリには有効ですが、業務固有の認可は依然としてアプリ内ロジックが必要になりやすいです。

### API ゲートウェイ適用

ゲートウェイは、認証、トークン検証、テナントルーティング、粗いレート制御、広域 API ポリシーに向きます。ただし機微な業務認可をそこだけに押し込むべきではありません。

### マルチテナント / B2B フェデレーションパターン

共有プラットフォームでは、テナント分離、委任管理、外部アイデンティティ連携、パートナーごとのポリシー上書きが必要です。この領域では ReBAC と範囲付き委譲が重要になります。

## 実装と運用

### パターン選択の基準

- 遅延の許容度
- 整合性要件
- チーム自律性のモデル
- 監査可能性 / 説明可能性の要求
- 信頼境界の位置
- ポリシー展開 / 巻き戻しを運用できる成熟度

### テナント連携とパートナー連携の運用

マルチテナント / B2B フェデレーションパターンは、テナントのアイデンティティ、パートナーのアイデンティティ、プラットフォーム運用者のアイデンティティを、データモデルとポリシー評価の両方で明確に分離すると安定します。実務上は、テナントスコープの識別子、issuer を意識したクレーム対応付け、テナントやパートナー境界に束縛された委任管理、home-realm discovery とパートナー受け入れ手順の明示が必要です。

この領域で ReBAC が重要になるのは、パートナーアクセスが単純なロール参照では済まないからです。たとえば、販売代理店は自分が紹介した顧客だけを管理できる、サプライヤーは自社の法人に紐づく購買注文だけを閲覧できる、テナント管理者は招待はできても他テナントへ越境できない、といった関係を表現する必要があります。

### 実務的な推奨

大規模システムの多くは最終的にハイブリッドになります。中央でポリシー意味論を定義し、適用は分散し、業務不変条件はアプリが確認します。重要なのは純粋性ではなく、責任分界を明示することです。
