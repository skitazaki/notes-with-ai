人間以外の主体に対するアイデンティティ管理は、アクセス制御アーキテクチャの中でも急速に重要性を増している領域です。現代のシステムには、人間の運用者よりもはるかに多くのサービス、ワークロード、パイプライン、デバイス、エージェントが存在します。

## 概要

この領域では、3 つの用語を分けて扱う必要があります。非人間 ID（non-human identity）は、人間以外の ID 全体をどう管理するかという包括的な考え方です。マシン ID（machine identity）は、サービス、デバイス、ジョブ、ボット、証明書、鍵、トークンなどを含む機械の認証情報全般を指します。ワークロードアイデンティティ（workload identity）は、その中でも実行中の Pod、コンテナ、VM、関数、サービスのようなワークロードに結び付く動的 ID を指します。

従来は静的シークレットや長寿命サービスアカウントに依存しがちでしたが、現代では証明、短命資格情報、証明書ベースのアイデンティティ、厳密に絞られたサービス間認可が中心になります。この領域は、Kubernetes、サーバーレス、CI/CD、API、サービスメッシュ、ロボティックオートメーション、IoT にまたがります。課題はワークロードを認証することだけではありません。その実行役割に必要最小限の動的 ID と権能を与えつつ、より広い非人間 ID 全体を統制することです。

## 主要概念

### 用語の境界

- **非人間 ID（non-human identity）**: 人間以外の ID 全体を管理する考え方。サービス、ワークロード、CI/CD、ロボット、デバイス、エージェント、共有自動化主体まで含む
- **マシン ID（machine identity）**: 機械が使う認証情報全般。証明書、鍵、トークン、サービスアカウント、クラウドロール、デバイス証明書などが含まれる
- **ワークロードアイデンティティ（workload identity）**: 実行中ワークロードに結び付く動的 ID。Pod、コンテナ、VM、関数、ジョブ、実行中サービスのように、起動・停止・再配置とともに変わり得る主体に対する識別子

### 実務上のアイデンティティ形態

実務上は、マシン ID とワークロードアイデンティティは主に次の形で現れます。

- サービスアカウントやクラウドロール
- X.509 証明書と mTLS アイデンティティ
- ワークロードフェデレーション用の OIDC / OAuth トークン
- 署名付きワークロード証明
- ハードウェアに根差したデバイスアイデンティティ

### なぜ静的シークレットは破綻しやすいか

長寿命の共有シークレットは、影響範囲を広げ、属性把握を弱くし、ローテーションを困難にします。複数サービスが同じ資格情報を提示するため、どのワークロードが動いたのかも曖昧になります。これはマシン ID 全般の問題ですが、特に実行時文脈と結び付くワークロードアイデンティティでは致命的です。

証明、ローテーション、ワークロードメタデータに結び付いたポリシーチェックを伴う自動発行の短命資格情報は、封じ込めと可観測性を改善します。

### SPIFFE / SPIRE とワークロード信頼

ワークロード信頼とは、動いているソフトウェア実体を「どのネットワークにいるか」ではなく、「その実体が何者かを証明できるか」で信頼する考え方です。Kubernetes、マルチクラウド、オートスケール環境では IP アドレスや配置先が絶えず変わるため、IP ベース信頼だけでは破綻しやすくなります。

SPIFFE は仕様です。正式には Secure Production Identity Framework For Everyone といい、ワークロード ID をどう表現するかを標準化します。代表例は `spiffe://company.internal/payments/api` のような SPIFFE ID で、人間にとってのログイン ID に近い役割を、サービスや Pod や実行中ワークロードに与えるイメージです。

SPIRE は実装です。SPIFFE Runtime Environment と呼ばれ、ノードやワークロードを attestation し、本当にそのワークロードなのかを確かめたうえで、短命な身分証を発行します。ここで発行されるのは、mTLS で使う X.509 SVID や、トークン用途で使う JWT-SVID です。SVID は SPIFFE Verifiable Identity Document、つまり SPIFFE 形式の検証可能な身分証です。

実際の流れは、Pod やサービスが起動し、SPIRE が Node、Namespace、ServiceAccount、各種プラットフォーム属性などの実行時証拠を確認し、そのワークロードに SPIFFE ID を割り当て、短命な資格情報を自動ローテーションする、という形になります。通信相手はその資格情報を検証して、「どこから来たか」ではなく「誰なのか」に基づいて mTLS や認可判断を行えます。

このため SPIFFE / SPIRE は、Zero Trust、Service Mesh、Kubernetes security でよく登場します。Istio でも `spiffe://...` 形式の ID が内部で使われることが多く、SPIFFE は特定製品の細部というより、機械同士の信頼を支える共通のアイデンティティ基盤として理解するのが適切です。

公式リンク: [SPIFFE ホーム](https://spiffe.io/), [SPIFFE Overview / Specifications](https://spiffe.io/docs/latest/spiffe-about/overview/), [SPIRE Project](https://spiffe.io/spire/)

## 実装と運用

### 推奨パターン

- クラウドが対応しているならフェデレーテッドなワークロードアイデンティティを使う
- 広すぎるサービスアカウントを狭い実行時アイデンティティに置き換える
- pod、namespace、cluster、image signature、build provenance などの証拠にアイデンティティを結び付ける
- ワークロードの認証と、その操作の認可を分離する
- 資格情報は自動ローテーションし、発行異常を監視する

### SPIFFE / SPIRE とフェデレーションの実装

SPIFFE / SPIRE の価値が最も大きいのは、クラスタや実行基盤をまたいで持ち運べるワークロード ID が必要な場合です。プラットフォーム固有のサービスアカウント意味論だけに依存せず、attestation に基づいて短命資格情報を発行し続けることで、侵害時の窓を狭くできます。

ワークロードフェデレーションは、証明書だけでなく OIDC / OAuth トークンでも実現できます。たとえば、あるクラウドや CI システムが別環境で ID を引き受けるケースです。運用上の論点は「証明書かトークンか」そのものより、発行が短命で、audience に束縛され、検証可能な実行時証拠に結び付いているかです。Service Mesh を併用する環境では、SPIFFE 互換 ID が mTLS、ポリシー評価、テレメトリ相関を揃える共通層になることも多くあります。

### 高リスク領域

- 広い本番権限を持つ CI/CD システム
- 複数業務で使い回される共有ロボット / バッチアイデンティティ
- 期限切れしない API トークン
- 過剰権限ワークロードにマウントされた Kubernetes サービスアカウント
- 失効や証明の仕組みを持たないデバイス群

運用上の目標は、機械の認証情報管理をシークレット配布の問題から、アイデンティティとポリシーの問題へ変えることです。その際、マシン ID、ワークロードアイデンティティ、非人間 ID を混同せず、どの層の制御を議論しているのかを明確にする必要があります。
