データ中心の組織において、データプロダクトは単なるデータセットではありません。明確なインターフェース、オーナーシップ、信頼性期待値、ライフサイクルを持つ消費可能な資産です。

## 概要

データセットをデータプロダクトに変えるのはメタデータです。メタデータは、そのプロダクトが何を意味し、誰が責任を持ち、どのように使うべきで、どの程度の品質や鮮度を期待でき、変更をどう扱うかという境界を表現します。

このメタデータ面がなければ、技術的には有用なデータを公開しても、再利用可能で信頼できるプロダクトにはなりません。データプロダクトには、人にもプラットフォームにも扱えるだけの明示的な契約、版管理ルール、サービス期待値が必要です。

## 基本概念

### プロダクトインターフェースとしてのメタデータ

最低限、利用可能なデータプロダクトには次のメタデータが必要です。

- 業務目的と想定利用者
- オーナーおよびスチュワードの連絡先
- スキーマとインターフェースの約束事
- 鮮度、可用性、品質に関する SLA または SLO
- 分類、利用制約、ライフサイクル状態

このメタデータは API 契約に近い役割を果たします。利用者は暗黙知に頼らず、そのプロダクトに安全に依存できるかを判断できるべきです。

### データ契約

データ契約は、提供元と利用者の期待を明示します。具体的には次を含みます。

- スキーマ形状と互換性ルール
- 意味上の前提と指標定義
- 鮮度ウィンドウと配信頻度
- バリデーションルールと品質閾値
- 廃止手順と移行手順

契約は変化をなくすものではなく、変化を統制可能で説明可能にするものです。

### 版管理と進化

強いプロダクトモデルでは、追加的進化と破壊的変更を区別します。実務では、どのスキーマ変更、意味変更、ポリシー変更が通知、並行公開、明示的移行支援を必要とするかを定義すべきです。

ここでもメタデータが運用的に効きます。バージョン識別子、依存グラフ、下流利用状況、廃止状態があれば、プラットフォームは場当たり的ではなく計画的に変更を扱えます。

## プロダクトインターフェースとしてのメタデータ

データプロダクトで最も重要なのは、メタデータが利用者にとってのインターフェースになることです。目的、オーナー、契約、品質期待値、ライフサイクルが明示されて初めて、他チームはその資産に安全に依存できます。

### インターフェースに含まれるもの

これは、カタログに説明文を追加するだけの話ではありません。プロダクトメタデータは、その資産が何を意味し、どの程度安定し、どの水準のサービスを期待でき、変更をどう伝えるかを示します。そのため、利用者は毎回ゼロから期待値をすり合わせずに再利用できます。

### 提供側に求めること

メタデータをインターフェースとして扱うと、提供側の責務も変わります。提供側は API と同じように、互換性、廃止、品質を継続的に管理しなければなりません。これによって、プロダクト境界は曖昧なものではなく、維持される約束事になります。

### 契約が信頼を支える

この意味で、契約、SLO、バージョン情報は単なる事務負担ではありません。ローカルなデータ公開を、他チームが信頼して使えるプロダクトへ変える仕組みです。

## 実装と運用

### 実践的な指針

- 公開するすべてのデータプロダクトに最低限のプロダクトテンプレートを用意する
- SLA、オーナー、ライフサイクルのメタデータを任意記述ではなく必須項目として扱う
- 可能な限り契約チェックを CI/CD、取り込み、リリースの流れに組み込む
- 破壊的変更前に影響分析できるよう、下流利用者を追跡する

### よくあるアンチパターン

一つは、運用上の約束を伴わないまま、すべてのテーブルをデータプロダクトと呼ぶことです。もう一つは、契約を重くしすぎて公開速度を著しく落とすことです。良いプロダクトメタデータは、信頼を作るだけの厳密さを持ちつつ、現場が更新し続けられる軽さも必要です。

### 具体的な実装例

- **Apache Iceberg** は、スナップショットベースの版管理、スキーマ進化、パーティションメタデータを通じて、データプロダクト契約のテーブル実装として使えます。
- **OpenMetadata** や **DataHub** は、オーナー、SLA または SLO、下流依存、ライフサイクル状態を公開し、プロダクトインターフェースそのものを可視化できます。
- 基盤として **Open Table Format** を採用すると、保存契約を単一の計算エンジンから分離しやすくなり、分析スタックをまたいでもプロダクト境界を保ちやすくなります。
- **Open Semantic Interchange** は、スキーマだけでなく再利用可能な業務意味も一緒に運べるように、意味定義をツール間で交換する実装例として位置付けられます。

### なぜ重要か

データプロダクトは「プロダクトとして扱うデータ」と説明されますが、重要なのはプロダクトにはインターフェースが必要だという点です。メタデータがそのインターフェースを提供し、提供元には公開境界を、利用者には依存前に評価できる判断材料を与えます。
