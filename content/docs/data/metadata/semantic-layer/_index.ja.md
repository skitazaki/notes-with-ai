チームごとに同じ言葉を異なる意味で使い、業務ロジックがローカルのダッシュボードやパイプラインに閉じ込められている限り、メタデータはドメイン横断で拡張できません。

## 概要

セマンティックレイヤーは、業務上の意味を安定化させるメタデータアーキテクチャの部分です。顧客、注文、売上、リスク、在庫といった概念を、複数ツールや複数ドメインで再利用できる共有形式で表現できるようにします。

これは特にデータメッシュやセルフサービス環境で重要です。各ドメインには自律性が必要ですが、利用者はなお、指標比較、関連プロダクトの発見、ドメイン横断統合を可能にするだけの意味的一貫性を必要とします。

## 基本概念

### 共有言語とローカルな意味のバランス

ドメイン固有の言葉と全社적整合性の間には常に緊張があります。良いセマンティックアーキテクチャはローカルなニュアンスを消すのではなく、ローカルな意味と共有意味の関係を明示します。

典型的には次の層があります。

- **ビジネスグロッサリー**: 共通概念と定義を整理する
- **メトリクスレイヤー**: 計算式やディメンションロジックを再利用可能にする
- **オントロジー (Ontology) とタクソノミー (Taxonomy)**:
  - **オントロジー**: 概念間の複雑な関係性や意味的な制約、推論ルールを定義する体系です。「AはBの一種である」だけでなく「AとCは排他的である」といった高度な論理関係を記述し、データ間の深い繋がりを表現します。
  - **タクソノミー**: 概念を階層的（親子関係）に分類・整理するための体系です。例えば「製品 > 電子機器 > スマートフォン」のようなツリー構造で、情報をナビゲートしやすくするために用いられます。
- **技術マッピング**: 業務概念を物理スキーマやパイプラインへ結び付ける

### 指標断片化の防止

同じ概念がダッシュボード、ノートブック、API、モデルごとに異なる計算で表現されると、指標断片化が起きます。メタデータは次を保持することで、この問題を減らします。

- 計算式定義と除外条件
- 粒度と集計ルール
- 承認済みソースプロダクトや標準エンティティ
- オーナー情報とレビュー責任

このメタデータがなければ、分析上の不一致は技術問題に見えて、実際には意味問題であることが多くなります。

### 相互運用性のための標準

持続的な意味相互運用性が必要な組織では、しばしば次のような標準や形式知モデルを参照します。

- **ISO/IEC 11179**: データ要素定義とメタデータレジストリ
- **SKOS**: タクソノミーや概念体系
- **OWL**: より豊かなオントロジーモデリング
- **DCAT** と **RDF**: メタデータ共有とグラフ表現

これらはすべてのチームに必須ではありませんが、メタデータをツール、ドメイン、組織境界を越えて流通させる必要がある場面では価値が高まります。

## 実装と運用

### 実践的な指針

- まずは顧客、売上、プロダクト、リスク、アクティブユーザーなど価値の高い概念から始める
- 意味のオーナーシップを明示する。責任者のいない共有定義はすぐにずれる
- 業務定義と物理実装を分離しつつ、その対応関係は保持する
- カタログ、クエリツール、API、AI 向け検索経路からセマンティックメタデータを利用できるようにする

### よくあるアンチパターン

一つは、すべてのドメインを早い段階で単一の硬直した全社オントロジーへ押し込むことです。もう一つは、意味を完全にローカル任せにして、利用者の手作業で整合させようとすることです。持続的な相互運用性は、必要なところではローカル概念を保ち、比較や統制が必要なところでは共有概念を用いる多層構造から生まれます。

### なぜ今重要か

分析、アプリケーション、AI システムが同じデータ基盤に依存するほど、セマンティックメタデータの価値は高まります。再利用が曖昧さへ崩れるのを防ぐ仕組みが、それだからです。
